## 1. プロジェクトの一言定義

**非中央集権的協調が成立するための3性質(誘因整合性・耐戦略性・個人合理性)を、5層構造として設計し、層間の型接続の整合性を機械的に検証しつつ、実行時の安全性・収束性も実証する、アーキテクチャパターンとその検証方法論。**

(注: 圏論(DisCoPy)は型接続検証の実装手段の1つであり、検証力は限定的。5章の言語選定結論と同じ認識を、一言定義・対外的な説明にも適用する。`DECISIONS.md` D-08)

対象系: **戦略的多主体協調系(SMAS)**。中央の執行者を置かず、環境を介した間接的な作用で協調と逸脱修正を実現する。

**今のフェーズ**: 1ケース目(タスク配分ドメイン、VCGメカニズム)の簡易スクラッチ実装。目的は「疎通確認」であり、本番品質のシステムを作ることではない。

## 2. 絶対に守るべき設計原則(違反したら作業を止めて確認する)

1. **中央の執行者を作らない**。①環境層の状態は、必ず独立したストア(EnvironmentClient経由)に置く。フレームワーク(LangGraph等)のStateオブジェクトに、データの実体を直接持たせてはならない(参照プロキシパターン、詳細は7章)
2. **A/B構造を守る**: 各層の「型・インターフェース」(A側)と「具体的な中身の実装」(B側)を、必ずファイル・モジュールとして分離する。A側は`schemas/`、B側は`engine/`等、テンプレート構成に従う
3. **②誘因構造層だけが「都度の数学的導出」を要する層**。①③④⑤は、既存の共通実装・確立されたライブラリをそのまま使い、車輪の再発明をしない
4. **プラガブル性は「型の互換性」のみを意味する**。「振る舞いの同等性」(LLMが理論通りに動くこと)は保証されない、という前提でコードを書く。これを暗黙に仮定するコードを書かない
5. **圏論による検証は事後チェック**(実装後にDisCoPyで確認)であり、事前の理論保証ではない。評価対象は「層をまたぐ接続の整合性」であり「エンジン内部の数式の正しさ」ではない
6. **通信は同期または部分同期を前提とする**。完全非同期の実装は、優先権の非対称を意図せず持ち込むリスクがあるため避ける(詳細は`SMAS_theorymap.md` 1章)

## 3. スコープ(何を作り、何を作らないか)

### 対象とする
複数の自律的な主体が、限られた資源・情報を巡って、正直な行動が必要になる協調様式(資源配分、投票、委任・権限移譲、信用形成、市場マッチング等)

### 明示的な前提(解決せず"仮定する"と宣言する。詳細は`scope_exclusions_and_deferrals.md` Part 0)
参加者の合理性・参加者の同一性(Sybil/偽名入札耐性)・支払いの執行・同期ラウンド進行の権威・効用の準線形性は、**外生的に保証されると仮定する**。これらは中央性の残余であり、「完全な非中央集権」とは主張しない。

### 絶対に手を出さない(理由は`scope_exclusions_and_deferrals.md`参照)
- インフラの可用性・冗長化・耐障害性(ビザンチン耐性含む)
- 独自の通信プロトコル設計(MCP/A2A等の「配線」は前提とせず、JSON Schemaレベルの互換性のみ意識)
- 協力ゲーム理論(提携形成)、真にボトムアップな創発的ガバナンス
- LLMの内部推論品質そのものへの介入
- 隠れた行動(モラルハザード)への本格対応。`process_trace`という任意拡張ポイントの型だけ用意し、中身は実装しない

### 今回実装しない、将来の拡張候補
POMDP、構成的アプローチへの格上げ、平均場理論、パーコレーション理論。**指示なく先回りして実装しないこと。**(最適化ベースエージェントはD-34でケース1に実装済み)

## 4. アーキテクチャ: 5層構造(責務分離版をマスターとする)

コードのファイル分割は、以下の責務分離版に従う(データフロー版はプレゼン・説明用、`architecture_family_map.md`参照)。

| 責務層 | 内容 | 実装 |
|---|---|---|
| ストレージ層 | 痕跡の読み書き・減衰のみ | `environment.py`(共通実装、原則そのまま使う) |
| メカニズム定義層 | 配分ルール+支払いルールを不可分な1つの仕様として定義 | `cases/<ケース名>/incentive_engine.py`(**ここだけケースごとに書く**。例: `cases/task_allocation/incentive_engine.py`) |
| メカニズム実行層 | ストレージから読み取り、定義層のルールを計算・適用 | `aggregation.py`(共通実装。集約計算はファミリーに応じ選択: VCG系の配分決定=割当問題は`scipy`、投票系は`pref_voting`。詳細は`mechanism_catalog.md`結論の注記) |
| 主体決定層 | 各エージェントの意思決定ロジック | `agents/`(ルールベース→LLMモック→LLM実物の順に実装) |
| 構造検証層 | 合成則・可換性の検証 | `verification.py`(`DisCoPy`利用) |
| (横断)アクセス制御 | 誰が何を読み書きできるか | 全層をまたぐPydanticバリデーション(プロセス内の型検証に留まる。本番実装での認証・認可の執行点は`scope_exclusions_and_deferrals.md` Part 0「壁の執行力の限界」参照) |

**①環境層の痕跡の中身(`Trace.payload`)は、②と同じA/B構造(ジェネリック型)にする**。「何でも入るdict」にしない。`process_trace: Optional[dict] = None`を型に含めておく(中身は実装しない)。

**リポジトリ構成(単一リポジトリ+`cases/`、`docs/DECISIONS.md` D-23)**: `environment.py`・`aggregation.py`・`verification.py`・`schemas/`・`agents/`・`verification_kit/`(montecarlo・mdp_convergenceのみ)はリポジトリルートに置く共通実装。ケースごとの内容(`incentive_engine.py`・`deviation_test.py`・`config.yaml`・`smoke_test.py`・`generate_results_summary.py`・`demo_llm_real.py`・`quint/`・`results/`)は`cases/<ケース名>/`配下にまとめる。共通部分の修正が必要になったら、この場所(リポジトリルート)を正典として直し、`cases/`配下の全ケースの`smoke_test.py`で後方互換性を確認する。旧来の3リポジトリ構成案(`docs/repository_structure.md`)は不採用。

## 5. 技術スタック(Python一本化、確定)

| 層 | ライブラリ |
|---|---|
| 環境層 | dict/JSON永続化(将来Redis)、Pydantic |
| 誘因構造層 | 自作(支払い関数)、検算に`pygambit`(結託耐性の検証、`verification_kit/gambit_collusion.py`、D-33でケース1に初適用) |
| 集約層 | ケース1(VCG)の配分決定(winner determination)は`scipy`の割当解。`pref_voting`は投票ファミリー用。収束確率に`pymdptoolbox`(1回性エンジンには適用不可、詳細は`SMAS_theorymap.md` 2.1節) |
| 実行主体層 | ルールベース自作→LLMモック→LLM実物→最適化ベース(`scipy.optimize`、`agents/optimization_based.py`、D-34でケース1に初適用)→将来`pomdp-py`/`PettingZoo` |
| 検証層 | `DisCoPy` |
| 検証キット(独立併走) | Quint(TLAモード)、モンテカルロ、`pygambit`(結託耐性) |
| フレームワーク | LangGraph(第一候補、参照プロキシパターン必須) |

`pip install` 時は `--break-system-packages` を付与する。

### 言語選定の最終結論: TypeScriptへの移行は不採用

TypeScript(Effect、XState等)をメイン言語に据える案を検討したが、不採用と結論した。理由:

1. **これまでの技術スタックの中核(`DisCoPy`、`pref_voting`、`pygambit`、`pymdptoolbox`)は、いずれもPython専用、あるいはPythonが最も充実したエコシステムであり、TypeScriptには相当するライブラリがほぼ存在しない**
2. TSの強み(非同期I/Oの強さ、XStateによる状態遷移の型安全性)は、SMASの設計思想とはむしろ逆方向。非同期通信は「優先権の非対称をネットワークレベルで持ち込むリスク」として警戒する対象(`SMAS_theorymap.md` 1章)であり、状態機械はSMAS本体の主役ではない(集約関数が主役)
3. **圏論(構造検証)は、検証層の中で最も検証負荷が軽く、実質的な検証力への寄与も限定的**(①〜⑤の型接続の事後チェックのみ)。以下の縮小の経緯を踏まえると、Effectの圏論的表現力の高さのために言語を変える動機は費用対効果として見合わない
   - 検証層は圏論のみではなく4サブコンポーネントの1つに格下げ済み
   - 圏論の検証は事後チェックであり、絶対的な証明ではなくツールへの相対的な信頼と明記済み
   - 構成的アプローチ(型で縛る、圏論の本領)はPythonの言語限界で断念、検証的アプローチのみ採用と確定済み
   - メカニズムカタログのうち圏論の合成可能性が強く効くのは委任連鎖のみで、1回性のVCGにはほぼ効かないと確認済み
   - **実質的な検証力は、③頑健性=ゲーム理論・Gambit・モンテカルロ、②収束性=MDPが担っており、圏論は①〜⑤層の"型の接続"という限定範囲のみ**

**将来の限定的な再検討条件**: 「⑤検証層の構成的アプローチへの格上げ」を本気で目指す段階になった場合のみ、検証層に限定してTypeScript(Effect)への部分移行を再検討する余地がある。ただしこれは2ケース目以降、かつ1ケース目のスクラッチ完走後の話であり、**今回のスクラッチでは一切検討しない**。

### Julia(メイン/フル/部分ハイブリッド)も同様に不採用

Python×Juliaハイブリッド、フルJulia実装、Juliaメイン+Python部分利用のいずれも不採用と結論した。理由:

- 技術スタックの中核(`pref_voting`、`pygambit`)はPython(相当)でしか存在せず、Juliaに成熟した代替ライブラリがない
- Juliaが強い領域(数値計算の高速性、微分可能プログラミング、大規模シミュレーション、`Agents.jl`等)は、SMASの核心(誘因構造・社会選択理論・圏論的検証)とことごとくズレている。TypeScriptが持っていた周辺的な強み(フロントエンド型共有等)すら持たない
- 「Juliaメイン+Python部分利用」は、SMASの核心部分をライブラリの薄いJuliaで書き直し、本来充実しているPythonライブラリを"部分呼び出し先"に格下げする、最も非効率な組み合わせになる

**唯一の限定的な将来シナリオ**: 平均場理論(`aggregation_functions_catalog.md`の拡張候補、エージェント数が数百〜数万規模になった場合)を本格実装する段階でのみ、③集約層の大規模数値計算部分だけを、独立した計算ワーカー(`Agents.jl`等)として切り出す構成を再検討する余地がある。純粋な数値計算のみを担わせる限定的な役割であれば、TypeScript案が抱えていた「ロジックと計算の境界が引けない」問題は起きにくい。ただし2ケース目以降の話であり、**今回のスクラッチでは一切検討しない**。

## 7. LangGraph統合ルール(必須チェックリスト)

LangGraphのStateを使う場合、必ず以下を満たす。

```python
# NG
class State(TypedDict):
    agent_data: dict  # データの実体を直接持たせない

# OK
class State(TypedDict):
    environment_ref: EnvironmentClient  # ①環境層への参照のみ
```

コミット前に確認する3項目:
1. Stateはデータの実体ではなく、①環境層への参照のみを持っているか
2. 壁(アクセス制御)がフレームワークによって迂回されていないか
3. フレームワークのノード実行順序が、②③⑤の役割に越境していないか

## 9. デモシナリオ(3シーン構成、必ずこの順で実装・テストする)

```
シーン1(平常時): 正直申告 → 集約 → 配分決定
シーン2(逸脱注入): 1エージェントが過大申告(戦略的逸脱)
シーン3(自己拘束の確認): 同一ラウンド内で、逸脱エージェントの実現効用を
              正直申告時の効用(反実仮想)と比較し、「逸脱しても得をしない
              (むしろ損をする)」ことを数値で示す(耐戦略性の直接実証)
```

**旧シーン3(痕跡→信用低下→取引忌避という環境を介した間接修正)は、繰り返しゲームが主軸の2ケース目に移設した**(`DECISIONS.md` D-07)。理由: (a)1回性VCGでは逸脱は同一ラウンド内で自己懲罰的であり、評判による事後修正は理論上不要、(b)逸脱の「検出」は観測不可能な真の評価額を要し、スコープ外のモラルハザード対応と矛盾する。

シーン3の反実仮想比較は「計測・ログ出力」として実装し、メカニズムの構成要素(ジャッジ)にしないこと。コード上に「隠れた中央集権」(特定の1関数が全員の運命を一方的に決める構造)が紛れ込んでいないかの目視確認は、引き続き全シーンで行う。

## 10. 評価の測り方(25項目ではなく5大指標を主役にする)

レポート・ログ出力は、以下の5大指標を主役にする。25項目の評価観点は「内訳・根拠」として、大指標の結果に1行併記する(省略しない)。

| 大指標 | 測定方法 |
|---|---|
| ①到達可能性 | Yes/No判定(個人合理性・権力集中の不在を根拠として1行記載) |
| ②収束性 | MDPの収束確率(1回性エンジンでは適用不可、モンテカルロで耐戦略性の経験的頑健性を代替評価) |
| ③頑健性 | モンテカルロ結果の要約(逸脱が得になったケース数) |
| ④資源コスト | 計算量・実行時間の概算 |
| ⑤検証可能性 | DisCoPy・Quintのチェック結果Pass/Fail |

## 11. コミット・PR規約

- 新しいメカニズム(②の中身)を追加する際は、`mechanism_catalog.md`の該当ファミリーに実装後追記する
- **設計判断の追加・変更は、新規ドキュメントを作らず`docs/DECISIONS.md`に1エントリ(5行以内)追記する**(append-only。詳細説明の新ファイル化は禁止)
- **1ケース目の実装完走まで、`docs/`配下の新規ファイル作成を禁止する**(`DECISIONS.md`への追記のみ可)。理由: ドキュメント量が想定実装量を一桁以上上回っている現状への処方箋(`DECISIONS.md` D-12)

## 12. 迷ったときの判断基準

- 「この処理はどの層に書くべきか」→ 4章の責務分離表を確認
- 「この機能は実装すべきか」→ 3章のスコープを確認。スコープ外なら実装しない
- 「このライブラリを使うべきか」→ 5章の技術スタックを確認。表にないライブラリを追加する前に、既存ツールで代替できないか検討する
- 「これは新しい層が必要か」→ まず既存5層の拡張ポイント(`process_trace`等)で吸収できないか検討する。安易に新しい層を作らない
