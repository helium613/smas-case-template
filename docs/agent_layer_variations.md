# 実行主体層(④) バリエーション対応表

## 前提: 共通インターフェース

すべてのバリエーションは、以下のインターフェースを満たす限り、互いに差し替え可能(プラガブル性)。

```python
def decide(observation: ObservationInput) -> ActionOutput:
    ...
```

- **ObservationInput**: ①環境層の痕跡から取得できる情報(共通最小限データ + 実装固有の追加コンテキスト、オプショナル)
- **ActionOutput**: JSON Schemaで構造化された行動(`action`, `declared_value`等 + `reasoning`はオプショナル)

**重要な限界**: プラガブル性は「構造的な差し替え可能性(型の一致)」を保証するのみで、「振る舞いの同等性(理論通りに動くか)」は実装ごとに異なる。ルールベースは決め打ちなので常に理論通り動くが、LLMは言語的推論で判断するため理論からの逸脱があり得る。

---

## バリエーション一覧

| # | 実装パターン | 意思決定の中身 | ②誘因構造層への依存度 | 観測情報の粒度 | 実装コスト | 優先度 |
|---|---|---|---|---|---|---|
| 1 | **ルールベース** | 固定的な条件分岐(if-then)。「常に正直に申告する」等、決め打ち | 依存なし(ルールが独立して定義される) | 単純な数値のみで足りる | 低 | **1(最優先・基準点)** |
| 2 | **LLMモック(確率分布)** | `random.random() < p`のような、確率的に逸脱するダミー関数。JSON Schema互換の出力を返す | 依存なし | 単純な数値+確率パラメータ | 低 | **2(モンテカルロ用、必須)** |
| 3 | **LLM実物** | Anthropic API、Tool Use/JSON Schemaで構造化出力を強制。自然言語での推論を経て判断 | 弱い依存(誘因構造の中身を"理解"しているとは限らない、言語的推論のみ) | テキスト化された対話履歴・痕跡の文脈が必要 | 中(プロンプト設計) | **3(目玉シーンのみ数回)** |
| 4 | **最適化ベース** | 目的関数(誘因構造エンジンの支払い関数)を都度計算し、利得を最大化する行動を導出 | **強い依存**(②のルールを直接参照して最適反応を計算するため、②を差し替えると④の実装も書き直しが必要になりうる) | 誘因構造エンジンの支払い関数そのものが必要 | 中(`scipy.optimize`等) | 4(拡張、②の理論的性質を厳密に検証したい時) |
| 5 | **POMDP** | 部分観測の信念状態から、確率的に最適行動を推論(`pomdp-py`) | 中程度の依存(誘因構造を信念更新の報酬シグナルとして使う) | 独自の内部表現(信念状態)が必要 | 高 | 5(拡張、情報の非対称性を主題化したい時) |
| 6 | **信念空間+効用空間推定(高度な発展形)** | 不完全情報下でベイズ的に相手の価値観を推定し、モンテカルロ木探索で戦略を進化させる(富士通型の事例) | POMDPと最適化の中間的な依存 | 対話履歴・信念空間・効用空間という、最も豊富な観測情報が必要 | 非常に高 | 拡張候補(2ケース目以降、④のプラガブル性の"限界事例"として実証する用途) |

## 逸脱モデルの2種類: 戦略的逸脱と、合理性を欠いた逸脱

これまでの逸脱注入テストは、**「合理的な計算に基づく、戦略的な逸脱」**(過大申告等)を想定していた。しかし、LLM実物の場合、**ハルシネーション・プロンプトインジェクションへの脆弱性・文脈への過剰適応による、"論理的に破綻した逸脱"**という、別種の逸脱が起こりうる。確率分布モック(`llm_mock_design_guide.md`)は前者(戦略的逸脱)の近似としては機能するが、後者(合理性を欠いた逸脱)の近似にはならない。

**対応**: 逸脱注入テスト(`cases/<ケース名>/deviation_test.py`)に、**「敵対的LLM(Red Team Agent): "いかにしてシステムのルールを曲解し、自身の利益を不当に最大化するか"だけを指示された、攻撃的なLLM実物」**をシナリオの1つとして追加する。これは「LLM実物(優先度3)」の拡張として位置づけられ、②誘因構造層が"合理的な逸脱"だけでなく"予測不能な逸脱"に対してもどこまで頑健かを、実際のLLMで確認する。優先度は2ケース目以降(評価観点12番「適応的逸脱への頑健性」の精緻化)。

## 「観測情報の粒度が非対称」という設計課題への対処

前回確認した通り、ルールベースとLLMでは必要とする観測情報の形が大きく異なる。これを吸収するため、`ObservationInput`は以下の2段構造とする。

```python
class ObservationInput(BaseModel):
    # 共通最小限データ(すべての実装パターンが受け取れる)
    trace_summary: dict  # 数値・カテゴリで表現された痕跡の要約

    # 実装固有の追加コンテキスト(オプショナル)
    extended_context: Optional[str] = None  # LLM向け: テキスト化された文脈
    belief_state: Optional[dict] = None      # POMDP向け: 独自の内部表現
```

## 「行動の表現力が非対称」という設計課題への対処

`reasoning`フィールド(LLMのみが持つ、自然言語の理由)は、可視化層側で以下のフォールバック処理を行う。

```python
if action_output.reasoning:
    display(action_output.reasoning)
else:
    display("内部ロジックによる判断")  # ルールベース・最適化ベース向け
```

## モック/実物の使い分けマトリクス

| 用途 | 使うべき実装 | 理由 |
|---|---|---|
| 5層構造の疎通確認(1ケース目の最小構成) | ルールベース | 予測可能、他層のバグと切り分けやすい |
| モンテカルロによる耐戦略性の統計的検証 | LLMモック | 大量試行が必要(数万回規模になりうる)。実LLM課金・レイテンシは非現実的 |
| デモの説得力(「本物のAIエージェントでも動く」という実演) | LLM実物 | 数回〜十数回に限定。目玉シーンのみ |
| 検証キット(Quint/MDP)での理論検証 | ルールベース or LLMモック | 検証したいのは①〜⑤層の性質であり、LLM特有のノイズを混入させると何を検証しているか分からなくなる |
| ②誘因構造層の理論的性質(支配戦略均衡)の厳密確認 | 最適化ベース | "理論を理解した合理的エージェント"として振る舞うため、エンジン設計の正しさを最も厳密に検証できる |

---

## 実装優先順位(再掲・確定)

```
1ケース目に必要な最小構成:
  ① ルールベース(基準点)
  ② LLMモック(確率分布、大量試行用)
  ③ LLM実物(目玉シーン、数回のみ)

  → この3つで「動く・検証できる・見せられる」の3条件が成立する

2ケース目以降・精緻化フェーズ:
  ④ 最適化ベース(理論的厳密性を上げたい時)
  ⑤ POMDP(情報の非対称性を主題化したい時)
  ⑥ 信念空間+効用空間推定(④実行主体層のプラガブル性の"限界事例"を示したい時)
```
