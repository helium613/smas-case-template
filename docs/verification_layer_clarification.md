# ⑤検証層の再定義と、圏論導入モチベーションの整理

## 1. 背景: 「検証層」という名前への違和感

「検証層」という名前だけでは、以下2つの問題があった。

1. **何を検証するのかが名前から伝わらない**。「検証」という一般語と、「圏論・合成則・関手・可換性」という特殊な中身の間にギャップがある
2. **「検証層 = 圏論のみ」という誤解を招く**。実際には並行安全性(Quint)、耐戦略性の経験的検証(モンテカルロ/Gambit)、収束性(MDP)も「検証」の一部だが、これらは従来「検証キット」として⑤層の外側に切り離されていた

## 2. 再定義の方針(採用: 選択肢B)

⑤検証層を「圏論単体」ではなく、**複数の検証手段を束ねる層**として正式に再定義する。名前は「検証層(Verification Layer)」のまま維持し、範囲を実態に合わせて拡張する。

```
⑤検証層(Verification Layer)
  ├─ 構造検証(Structural Verification): 圏論・DisCoPy
  │    → 合成則(結合律・単位律)、可換性
  ├─ 動的安全性検証(Dynamic Safety Verification): Quint(TLAモード)
  │    → 並行性、競合状態、打ち切り時の状態遷移の安全性
  ├─ 実証的検証(Empirical Verification): モンテカルロ / Gambit
  │    → 耐戦略性の経験的頑健性(1回性エンジンでは「様々な入力パターンで
  │       逸脱が得にならないか」を検証。繰り返しエンジンでは収束確率とセット)
  └─ 収束検証(Convergence Verification): MDP(pymdptoolbox)
       → 繰り返し試行による、安定状態への収束確率・速度
```

**実装上の扱い**: 検証キット(Quint/MDP)は引き続き独立したモジュール・フォルダ(`verification_kit/`)として実装してよい。今回の変更は「概念上どこに属するかという説明ラベル」の変更であり、実装のフォルダ構成を変える必要はない。

## 3. 圏論(構造検証)が「いつ」「何を」検証するかの明確化

### 3.1 タイミング: 事後検証である(事前理論保証ではない)

```
1. ①〜④層を、schemas/の型定義に従って実装する
2. 実装完成後(あるいはCI/CD時)に、DisCoPyで
   「入出力の型をTy/Boxとして再構成し、合成則が成立するか」を計算する
3. 「成立する」なら安心材料、「成立しない」なら実装のどこかに矛盾がある、
   と事後的に発見する
```

- 「証明してから作る」のではなく「作ってから、壊れていないか確認する」
- 完全な「構成的アプローチ」(型で縛り、そもそも矛盾を書けなくする)は、Pythonの言語的限界(動的型付け、型ヒントは補助的チェックに留まる)により不可能と結論済み。検証的アプローチで進める方針を確定済み
- **準構成的アプローチ**(型で縛れる範囲を部分的に広げる)は、2ケース目以降の拡張候補として保留。ただし「完全な保証」ではなく部分的な改善に留まること、Pydantic(Python型システム)とDisCoPy(圏論型システム)という2つの異質な型システムの橋渡しコストは準構成的アプローチでも解消されないこと、の2点は残存する制約として明示しておく

### 3.4 関手による新規ケース対応づけの判定基準(ケース1の時点で確定しておく)

- 「2ケース目が既存の骨格に収まるか」の判定は、**「2ケース目の①環境層・②誘因構造層それぞれの型定義が、`smas-architecture/schemas/`の型を満たしているか」という機械的なチェックのみで行う**。この基準を`smas-case-template`のREADMEにチェックリストとして明記する
- **委任連鎖(A→B→C)の結合律検証は、ケース1(1回性・単一段階のVCG)では原理的に検証不可能**。委任連鎖が本質的に存在するのはケース2(信用枠配分/委任)以降のため、この検証はケース2実装時に初めて行う、という前提を明示しておく

### 3.2 確からしさの性質: 「理論保証」でも「確率的実証」でもない、中間的な位置づけ

| | 理論保証(証明) | 実証(経験的確認、モンテカルロ等) | 圏論による検証 |
|---|---|---|---|
| 対象 | すべての可能な入力に対する一般的な数学的成立 | 具体的サンプルに対する統計的確認 | **実装された、特定の1つの構造に対する厳密な確認** |
| 確からしさ | 100%(証明が正しければ) | 確率的(サンプル数依存) | **100%(実装された構造の範囲内では)。ただし"実装が正しく圏論の対象に変換されているか"という変換の正しさに依存** |

「一般的な定理の証明」と「個別のコードのテスト」の中間に位置する、独特な検証手段。

### 3.3 評価対象: 誘因構造エンジンの「境界」であり「中身」ではない

```
②誘因構造エンジンの「確からしさ」

  ├─ 中身(数式が耐戦略性を満たすか)
  │    → 数式の証明(既存定理の引用)・Gambit・モンテカルロが担当
  │      (圏論の検証範囲【外】)
  │
  └─ 境界(他の層と、入出力の型として正しく噛み合っているか)
       → 圏論(DisCoPy)の検証範囲【内】
```

圏論が検出するのは「個々の層は単体で見れば正しく動くが、層をまたぐ接続部分で静かに矛盾が生じている」というパターン。具体例:

| 検出できるケース | 内容 |
|---|---|
| ①環境層の壁とエンジンの入力要求の不一致 | エンジンが必要とするデータが、壁の設定変更により取得できなくなる |
| 委任連鎖(A→B→C)での結合律の破れ | 経路によって結果が変わってしまう(A→B→CとA→C直接で不一致) |
| 新規ケースでの型の不一致 | 環境層に新しい種類の痕跡を追加した際、集約層側が対応できていない |
| エンジン差し替え時の型ミスマッチ | 新しい誘因構造エンジンの出力形式が、既存の他層の期待と食い違う |

**注記**: 検証しているのは「層をまたぐ、入出力の型の連鎖(データフローの整合性)」であり、オブジェクト指向的な「クラス構造」ではない。対象(Ty)・射(Box)という圏論の語彙で捉えるべき。

## 4. 圏論導入モチベーションの妥当性確認

### 4.1 「ゲーム理論×圏論」は学術的に確立された組み合わせ

**Compositional Game Theory(Open Games)** という、ゲーム理論的構造(戦略・利得)を圏論のlens/opticで「合成可能な形」に表現する研究分野が既に存在する。ゲーム理論(特にメカニズムデザイン)は「複数プレイヤーの戦略・利得をどう合成して1つの結果に導くか」という問題そのものであり、「合成」が核心に内在するため、圏論との相性は必然的。

### 4.2 「システム構造の圏論化」はゲーム理論に限らない普遍的ニーズ

Categorical Cybernetics(フィードバックループ)、Applied Category Theory全般(化学反応ネットワーク、電気回路、データベーススキーマ、構成的意味論など)、**「合成可能性(Compositionality)が本質的に重要な問題を持つ分野」であれば広く応用されている**。SMASは「戦略的マルチエージェントの協調」という、合成可能性が本質的な問題を扱うため、圏論との相性が良い。

### 4.3 技術的興味だけでなく、具体的なモチベーションから出発している

| モチベーション | 対応する具体的な問題 |
|---|---|
| 委任連鎖で誘因構造が変質していないか検出したい | 結合律の検証 |
| 層をまたぐ接続で静かに矛盾が生じていないか検出したい | 4つの具体例(3.3節) |
| エンジン差し替え時も整合性を保証したい | プラガブル性の裏付け |
| 新規ケースが既存骨格に本当に収まるか機械的に確認したい | 関手による対応づけ |

**結論**: "面白そうだから使った"のではなく、実務上の具体的な失敗パターンから逆算して選ばれた、かつ学術的にも妥当性が裏付けられた選択。ただし「横断的パターン抽出を好む」という個人の気質が、動機の一部として働いていたことも事実であり、これは動機の厚みを増す要素として肯定的に捉えてよい。
