# 実装フェーズの技術選定対象一覧(粗め・Ph3向け)

各層・各機能について、「どんな種類の技術を選ぶ必要があるか」という選定"カテゴリ"の洗い出し。具体的な製品名は候補の一例であり、確定ではない。

---

## ①環境層: 痕跡ストア・共有バス

| 選定項目 | 選択肢のカテゴリ | 候補例(参考) |
|---|---|---|
| 痕跡の永続化ストア | KVS / イベントソーシング型DB / 分散台帳 | Redis、KurrentDB(旧EventStoreDB)、単純にPostgreSQL |
| エージェント間の通知・伝播 | メッセージブローカー / Pub-Sub | NATS(JetStream)、Redis Pub/Sub、Kafka |
| 状態合成の技法(束/CRDT) | CRDT実装ライブラリ / 自前実装 | Automerge、Yjs(JS系が主流、Python実装は限られる。自前実装も検討対象) |
| 壁(アクセス制御)の実装場所 | アプリケーション層で制御 / DB側のRow Level Security等 | Pydanticでのバリデーション(アプリ層)+必要ならDB側のACL機能併用 |

## ②誘因構造層: 支払い・エンジンの実行基盤

| 選定項目 | 選択肢のカテゴリ | 候補例 |
|---|---|---|
| 支払い・報酬の記録方式 | 分散台帳(オンチェーン) / オフチェーン台帳+定期精算 / 単純なDB記録 | スマートコントラクト基盤(経済的資源型)、単純なPostgreSQL(検証目的なら十分) |
| 誘因構造エンジンの計算コスト設計 | 経済的資源型(ガス代等) / 計算コスト型(PoW・検証可能遅延関数) | 1ケース目は「計算コストを模擬する軽量な値」で十分、本番実装で要再検討 |

## ③集約層: 集約計算の実行基盤

| 選定項目 | 選択肢のカテゴリ | 候補例 |
|---|---|---|
| 集約関数の実行環境 | 単純な同期関数呼び出し / 分散処理基盤 | 1ケース目は単純呼び出しで十分。大規模化(平均場理論が必要な規模)なら分散処理基盤(Ray等)を検討 |

## ④実行主体層: エージェント実行基盤

| 選定項目 | 選択肢のカテゴリ | 候補例 |
|---|---|---|
| エージェントのオーケストレーションSDK | グラフベース / 役割ベース / モデル駆動ループ | LangGraph(第一候補、参照プロキシパターン採用)、CrewAI・Anthropic Managed Agents等は見送り理由あり(既存議論参照) |
| LLM呼び出し基盤 | 直接API呼び出し / SDK経由 | Anthropic API(Tool Use/JSON Schema) |
| エージェント間通信規格(将来のA2A互換) | 標準プロトコル / 独自形式 | A2A(前提とはしないが、JSON Schemaレベルで互換性を意識) |

## ⑤検証層: 検証・型システム基盤

| 選定項目 | 選択肢のカテゴリ | 候補例 |
|---|---|---|
| 圏論表現ライブラリ | Python実装 / Julia実装 | DisCoPy(第一候補) |
| 並行安全性の形式検証 | モデル検査ツール | Quint(TLAモード)、PRISM/Storm(見送り、理由は既存議論参照) |
| 実行時の型チェック | 静的型検査 / ランタイム検証 | mypy(静的)、Pydantic(ランタイム) |
| 収束確率の数値計算 | MDPソルバー | pymdptoolbox |
| 均衡計算の検算 | ゲーム理論計算ライブラリ | Gambit(pygambit) |
| 社会選択理論の集約計算 | 投票・マッチングライブラリ | pref_voting |

## 選定の優先順位(1ケース目のスクラッチ実装時点)

**ほとんどの項目は「最も軽量な選択肢で十分」というのが1ケース目の結論**。分散台帳・分散処理基盤・専用ポリシーエンジンのような重量級の技術は、いずれも「疎通確認」という今回の目的には過剰。

```
1ケース目の実用最小構成:
  ①痕跡ストア: dict/JSON(将来Redis)
  ②支払い記録: 単純なメモリ上の値
  ③集約計算: 単純な同期関数
  ④オーケストレーション: LangGraph
  ⑤検証: DisCoPy + Quint + mypy/Pydantic

  → いずれも「なぜこれが十分か」の判断基準は、
    以前確定した「アーキテクトとしての説得力で十分、
    研究者・本番運用レベルの完全性は求めない」という方針に従う
```

本番運用を見据えた技術選定(分散台帳の要否、CRDTの本格実装等)は、②誘因構造層の「経済的資源型 vs 計算コスト型」という未決事項(`SMAS_theorymap.md`)の解消と連動するため、2ケース目以降で再検討する。
