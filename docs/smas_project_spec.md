## 0. プロジェクトの一言定義

**非中央集権的協調が成立するための3性質(誘因整合性・耐戦略性・個人合理性)を、5層構造として設計し、層間の型接続の整合性を機械的に検証しつつ、実行時の安全性・収束性も実証する、アーキテクチャパターンとその検証方法論。**

(注: 旧「4性質」表記は、①概念層を「中核3性質+前提背景+メタ理論」に再構成した際の更新漏れ。3性質表記に統一(`DECISIONS.md` D-09)。圏論の位置づけの降格はD-08)

対象系の名称: **戦略的多主体協調系(Strategic Multi-Agent System, SMAS)**
- 「戦略的」= 各主体が自己利益に基づき、嘘・逸脱をしうることを前提とする
- 中央の執行者(オーケストレーター)を置かず、環境を介した間接的な作用で協調と逸脱修正を実現する

## 1. スコープの明示(何を検証し、何を検証しないか)

### 対象とする
- 複数の自律的な主体が、限られた資源・情報を巡って、正直な行動が必要になる協調様式全般(資源配分、投票、委任・権限移譲、信用形成、市場マッチング等)

### 対象としない(意図的にスコープ外)
- インフラレベルの可用性・冗長化・耐障害性(ノード故障、ネットワーク分断)
- 通信プロトコル自体の設計(A2A、独自プロトコル等は「配線」として扱い、規格化しない)
- LLMの内部推論品質そのもの(思考の連鎖の質などは④実行主体層の中身の話であり、アーキテクチャ本体の関心事ではない)

## 2. 3階層モデル

| 階層 | 内容 | 普遍性 |
|---|---|---|
| **①概念層(Properties)** | 中核3性質(誘因整合性・耐戦略性・個人合理性)+前提背景(情報の非対称性)+メタ理論(必要多様性の法則、不完備契約理論) | 完全に普遍(既存数学) |
| **②骨格層(Structure)** | 5層構造。デザインパターンとして複数ケースに使い回す | ほぼ普遍(2ケース目で検証) |
| **③具体設計層(Instance)** | ケースごとの数式・アルゴリズム・パラメータ | ケース依存(消耗品、ただし再利用可能な「メカニズムのカタログ」を中間的に保持) |

### ①概念層の詳細構成
- **中核3性質(定理レベルの厳密さ)**: 誘因整合性、耐戦略性、個人合理性
- **前提背景**: 情報の非対称性(隠れた情報=Adverse Selection/隠れた行動=Moral Hazard の2種。後者は現状スコープ外、将来拡張候補)
- **メタ理論(比喩的援用、①全体を支える背景として注記)**: 必要多様性の法則(Ashby)、不完備契約理論(介入ポートの理論的裏付け)

### 図の使い分け方針
- **責務分離図(マスター)**: 実装が依拠する正確な構造。ストレージ層/メカニズム定義層/メカニズム実行層/主体決定層/構造検証層+横断的アクセス制御、という責務ベースの分割。内部設計用。
- **データフロー図(説明用)**: 環境→誘因構造→集約→実行主体→検証、という時間軸に沿った物語的な図。デモの説明で使用。ケース×シーン別の可視化は、静的な図を都度作るのではなく、共通ログフォーマットから生成する可視化装置を1つ作る方針。
- マスターは責務分離図、データフロー図はそこから派生的に単純化して生成する(逆はしない)。

## 3. 5層構造(②骨格層)

| 層 | 役割 | ケース間で共通/固有 |
|---|---|---|
| **①環境層 Environment** | 痕跡ストア、ルールの静的配置、壁(アクセス制御)、介入ポート、減衰関数 | 構造は共通、パラメータ(寿命・減衰率)はケース固有 |
| **②誘因構造層 Incentive Structure** | 正直な行動が支配戦略になるよう設計。**誘因構造エンジンを内包** | **中身(エンジン)は完全にケース固有** |
| **③集約層 Aggregation** | 決定論的な集約関数、操作耐性、局所計算・大域整合 | ほぼ共通(集約計算はファミリー依存: VCG系の配分決定=割当問題は`scipy`、投票系は`pref_voting`) |
| **④実行主体層 Agent** | プラガブルな意思決定ロジック(ルールベース/最適化/POMDP/LLM) | インターフェース共通、中身は差し替え可能 |
| **⑤検証層 Verification** | 合成則(結合律・単位律)の検証、関手による新ケース対応づけ | ほぼ共通(DisCoPyで実装) |

### 誘因構造エンジンの2階建て構造
- **A側(アーキテクチャ側)**: エンジンの型・規格(`payment_function`, `parameters`, `version`を持つインターフェース)。介入ポート(オフラインでのバージョン更新)を含む
- **B側(ケース実装側)**: 具体的な支払い関数(VCG等)。ケースごとに新規導出が必要な、唯一の「エンジン」部分

### ⑤検証層の拡張方針(検証的アプローチ→構成的アプローチ)
- **現状(1ケース目)**: 検証的アプローチ。①〜④を自由な形式で実装し、実装後にDisCoPyで合成則(結合律・単位律)を事後チェックする
- **将来の拡張(2ケース目以降で優先的に検討)**: **準構成的アプローチ**への部分的な格上げ。完全な構成的アプローチ(型のみで矛盾を防ぐ)はPythonの言語的限界(動的型付け、型ヒントは補助的チェックに留まる)により不可能と結論済み。実現できるのは「型で縛れる範囲を部分的に広げる」という準構成的な改善に限られる。`schemas/`の型定義にDisCoPyの語彙(Ty, Box)を部分的に持ち込むことは、新しい理論の学習ではなく既存ツールの応用のため、他の拡張候補(代数的位相幾何学等)より着手コストは低い
- **残存する懸念(未解消)**: Pydantic(Pythonの型システム)とDisCoPy(圏論の型システム)という、性質の異なる2つの型システムを橋渡しする実装コストは、準構成的アプローチにおいても解消されない。この橋渡し層の設計自体が、地道なソフトウェア工学上の作業として残る
- 位置づけ: 新しい理論の追加ではなく、既存ツール(DisCoPy)をより深く使う方向への格上げのため、他の拡張候補より学習コストが低い。ただし「言語限界」「二重運用コスト」という制約は解消されておらず、"完全な保証"は最初から目指さない、という前提を明示する
- 留意点: 検証層自体も「有限の道具」であり無謬ではない(DisCoPy自体のバグ等)。圏論による検証は絶対的な証明ではなく、確立されたツールを介した相対的な安心材料である、と正直に位置づける

## 4. 評価観点(22項目、詳細は evaluation_criteria.md 参照)

- ①概念層由来: 誘因整合性/耐戦略性/情報の非対称性の制御/必要多様性(4項目)
- ②誘因構造層由来: 結託耐性/コストリー・シグナリング/繰り返しゲームの安定性(3項目)
- ③集約層由来: 決定論性・局所-大域整合/操作耐性/公平性(3項目)
- ④実行主体層由来: プラガブル性/適応的逸脱への頑健性(2項目)
- ⑤検証層由来: 合成則の充足/権力集中の不在/関手性(3項目)
- 実証的観点: 並行安全性/収束性/通信密度の臨界点(3項目)
- 横断的観点: 委任連鎖の誘因伝播/逸脱注入からの回復力(2項目)
- 精緻化: 参加者の入退場への頑健性/開示情報の最小性(2項目)

### コア/拡張の区分(過剰設計を避けるため)
- **コア(1ケース目で必須)**: 誘因整合性、耐戦略性、決定論性・局所-大域整合、プラガブル性、合成則の充足、権力集中の不在、逸脱注入からの回復力
- **拡張(2ケース目以降・状況に応じて)**: 結託耐性、公平性、適応的逸脱への頑健性、通信密度の臨界点、参加者の入退場への頑健性、開示情報の最小性

## 5. 技術スタック(Python一本化)

3. LLM実物(Anthropic API、Tool Use/JSON Schema、目玉シーンのみ数回)
4. 最適化ベース(拡張)

## 6. スクラッチ実装のミニマル構成(今回の目的)

```
[1ケース目候補: タスク配分 or 委任・権限移譲(いずれか選定)]

① 環境層: dict + Pydantic(壁の型定義)
② 誘因構造層: VCG的支払い関数(自作、簡単な既知ケースで検算)
③ 集約層: VCGの配分決定は`scipy`の割当解(winner determination)。`pref_voting`は投票ファミリー採用時のみ
④ 実行主体層: ルールベース + LLMモック(確率分布)
⑤ 検証層: DisCoPy(合成則の最小チェック)

検証シナリオ(3シーン):
  シーン1(平常時): 正直申告→集約→配分決定
  シーン2(逸脱注入): 1エージェントが過大申告
  シーン3(自己拘束の確認): 同一ラウンド内で、逸脱エージェントの実現効用を
    正直申告時の効用(反実仮想)と比較し、「逸脱しても得をしない」ことを
    数値で示す(耐戦略性の直接実証)
  ※旧シーン3(痕跡→信用低下→取引忌避)は2ケース目(繰り返しゲーム)へ移設。
    理由はDECISIONS.md D-07(1回性VCGと評判動学の混在、検出の観測不可能性)
```

### 疎通確認で見るべきポイント
- ①〜⑤のインターフェースが実際に噛み合うか(型の齟齬がないか)
- ②誘因構造エンジンが、理論通り正直申告を最適にできているか(簡単な数値例で確認)
- シーン3の反実仮想比較で、逸脱が得にならないことが数値で確認できるか(比較は計測・ログとして実装し、メカニズム構成要素にしない。隠れた中央集権の目視確認は全シーンで継続)
- コア評価観点(上記7項目)が、この最小構成で一応測れる状態になっているか

### 7.1 フレームワーク統合ガイドライン(A側の設計項目として確定)

- **既存フレームワーク(LangGraph等)は④実行主体層の一実装オプションとして扱い、独自のSMAS用フレームワークは新規開発しない**(土管の再発明を避ける方針)
- **状態オブジェクトの薄いプロキシ化ルール**: フレームワークが提供する状態管理機能(LangGraphのState等)は、①環境層への参照(EnvironmentClient)のみを持つものとし、データの実体を直接持たせない
- **統合時の3項目チェックリスト**(`framework_binding_schema.py`で型として強制、`05_framework_integration_guideline.md`で文書化):
  1. 状態の実体は①環境層にあるか(フレームワークの状態オブジェクトは参照のみを持つか)
  2. 壁(アクセス制御)はフレームワークを迂回されていないか
  3. フレームワークのオーケストレーション(ノード実行順序)が、②③⑤の役割(誘因構造・集約・検証)に越境していないか
- **フレームワーク選定の適合度評価**: LangGraph(明示的な状態スキーマ定義により、プロキシ化ルールを強制しやすい)を第一候補とする。CrewAI・Managed Agents等は状態管理が内部化・委譲されており、プロキシ化の強制が難しいため見送り
- **1ケース目の実装候補**: LangGraphでの基盤実装を行い、上記ルールの実現可能性・適合度を検証する(デモ実装の一部として位置づけ)
