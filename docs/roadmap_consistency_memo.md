# ロードマップ整合メモ: 3層→5層の入れ子構造の復元、大指標と評価観点の関係整理

## 1. 問題の所在: 「3層→5層」の入れ子が、フラットな1本の5分類に潰れていた

`SMAS_roadmap.md` Ph2-1の表(概念層/誘因構造層/骨格層/データ構造層/検証層)は、**もともと3層構造(①概念層/②骨格層/③具体設計層)における「②骨格層」1段の中身(旧5層)を具体化しようとした結果、①③と横並びの、フラットな5要素として誤って展開されてしまったもの**。

## 2. 正しい入れ子構造(復元版)

```
①概念層(普遍)
  誘因整合性・耐戦略性・個人合理性
  + 前提背景(情報の非対称性)
  + メタ理論(必要多様性の法則、不完備契約理論)

②骨格層(SMASのアーキタイプとしての骨格、5要素で構成)
  1. 環境層(Environment)
  2. 誘因構造層(Incentive Structure)
  3. 集約層(Aggregation)  ※旧ロードマップの「骨格層(手続き構造)」はここに吸収
  4. 実行主体層(Agent)    ※旧ロードマップで行方不明になっていたが復元
  5. 検証層(Verification)

③具体設計層(ケース依存)
  各層の中身の実装(VCG、pref_voting、DisCoPy、LangGraph統合、
  CRDT/束構造等の状態合成技法 等)
```

### 「データ構造層(束/CRDT)」の位置づけ

新しい5つ目の層として独立させる必然性はない。**「①環境層の"具体設計"における、状態合成の技法(束/semilattice/CRDT)」という、③具体設計層側の選択肢**として位置づける。以前確定した「①環境層の構造は共通、パラメータ・実装技法はケース依存」という整理と一致する。

### 「実行主体層」の復元

プラガブル性(構造的差し替え可能性のみを保証、振る舞いの同等性は保証しない)、ルールベース/LLMモック/LLM実物/最適化ベース/POMDPという実装バリエーション、観測情報の粒度の非対称性への対応(`agent_layer_variations.md`参照)は、すべてこの層に属する。新ロードマップの5層表には存在しなかったが、②骨格層の4番目として復元する。

### 「骨格層」という名称の衝突への対処

3階層モデルの「②骨格層」(5層構造全体を指す)と、Ph2-1表で登場した「骨格層」(手続き構造を指す、5層の1つ)は、同じ単語で異なる対象を指しており、読者を混乱させる。**Ph2-1表側の「骨格層」は、上記の入れ子構造復元により「③集約層」に吸収されるため、この名称衝突は自然に解消される。**

## 3. 大指標(①到達可能性〜⑤検証可能性)と評価観点22+3項目の関係

両者は包含関係ではなく、**抽象度の異なる、別レイヤーの指標体系**。

| | 大指標(①到達可能性〜⑤検証可能性) | 評価観点22+3項目(`evaluation_criteria.md`) |
|---|---|---|
| **目的** | 設計段階で、どのサブタイプ(3パターン)を選ぶかを判断するための、粗い羅針盤 | 実装段階で、個々の層・個々のケースが具体的に正しく機能しているかを確認するための、詳細なチェックリスト |
| **抽象度** | 高い(理論的性質のカテゴリ) | 低い(実装可能な、個別の確認項目) |
| **使うタイミング** | Ph2(設計、サブタイプ分岐の判断) | Ph4(実証、実際に測定) |

### 対応表(大指標 → 評価観点)

| 大指標 | 対応する評価観点(22+3項目から) |
|---|---|
| ①到達可能性 | 個人合理性(25番、注記参照)、権力集中の不在(14番) |
| ②収束性 | 決定論性・局所-大域整合(8番)、逸脱注入からの回復力(20番)、収束性(17番) |
| ③頑健性 | 誘因整合性(1番)、耐戦略性(2番)、結託耐性(5番)、適応的逸脱への頑健性(12番)、打ち切り耐性(23番) |
| ④資源コスト | **既存の評価観点22+3項目に明示的な対応なし。新規追加が必要**(誘因構造エンジンの計算コスト、ガス代、通信コスト等) |
| ⑤検証可能性 | 合成則の充足(13番)、関手による対応づけ(15番)、並行安全性(16番) |

### 結論

大指標は「Ph2で、3つのサブタイプのどれを選ぶかを判断する、上位の羅針盤」、評価観点22+3は「Ph4で、選んだサブタイプが実際に機能するかを測る、下位の物差し」という、階層関係で整合が取れる。**④資源コストのみ、既存の評価観点リストに対応する項目がないため、`evaluation_criteria.md`に新規項目として追加すべき。**

**注記(1ケース目完走後の振り返りで発覚、`DECISIONS.md` D-20)**: 上表の「個人合理性(3番)」という参照は誤りだった。`evaluation_criteria.md`の3番は実際には「情報の非対称性の制御」であり、個人合理性(①概念層の中核3性質の1つ)には当初番号が割り当てられていなかった。`evaluation_criteria.md`に25番として追加し、本表・下記運用イメージの参照を25番に修正した。

### 3.1 運用方針: 大指標を主役、評価観点24項目は内訳として扱う(検証損失なしの圧縮)

**問題意識**: 大指標(5つ)と評価観点(24項目)を並行して"両方測るべきもの"として扱うと、実質24項目前後を同時に追いかけることになり、1ケース目の疎通確認としては過剰になる。

**方針**: 評価観点24項目は削除・省略するのではなく、**「大指標を測る際の内訳・根拠」という位置づけに格下げする**。確認作業そのものは変えず、レポート・記録の単位だけを24から5に圧縮する。

```
[運用イメージ]
①到達可能性: Yes/No(根拠: 個人合理性[25]、権力集中の不在[14]を確認)
②収束性: MDPの収束確率(根拠: 決定論性・局所-大域整合[8]、
          逸脱注入からの回復力[20]、収束性そのもの[17])
③頑健性: モンテカルロ結果の要約(根拠: 誘因整合性[1]、耐戦略性[2]、
          結託耐性[5]、適応的逸脱への頑健性[12]、打ち切り耐性[23])
④資源コスト: 計算量・実行時間の概算(根拠: 資源コスト[24])
⑤検証可能性: DisCoPy・Quintのチェック結果Pass/Fail(根拠: 合成則の充足[13]、
          関手による対応づけ[15]、並行安全性[16])
```

**検証損失について**: 確認作業自体は24項目すべてに対して引き続き行うため、実質的な検証の損失はない。**唯一のリスクは、大指標への要約時に内訳確認が省略される運用上の手抜き**であり、これを防ぐため「大指標の結果を報告する際は、必ず内訳のどの項目を・どう確認したかを1行併記する」というルールを`smas-case-template`のREADMEに明記する。

## 5. Ph2-1表の指標対応における重複の明記(懸念点5)

`SMAS_roadmap.md` Ph2-1の表で、「骨格層」「データ構造層」がともに②収束性に対応づけられているが、区別が明記されていない。上記2節の入れ子構造復元により「骨格層」は③集約層に吸収されるため、この重複自体は解消される。ただし、もし将来的に「手続きの収束(集約層が扱う、合意への収束)」と「状態の収束(束構造・CRDTが扱う、データレプリカ間の収束)」を明示的に区別したい場合は、注記を1行加えるとよい:「③集約層は"手続き上の合意への収束"、①環境層の状態合成技法(CRDT等)は"データ状態としての収束"を扱う。両者は"収束性"という同じ大指標②の下位で、対象が異なる」。
