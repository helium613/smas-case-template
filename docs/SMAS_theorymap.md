## 1. 通信モデルの選択肢と、①到達可能性への影響(新規検討)

### 1.1 分散システム理論における3つの標準モデル

| モデル | 定義 | 到達可能性への含意 |
|---|---|---|
| **同期(Synchronous)** | メッセージ配送時間・処理速度に既知の上限がある。「時間内に応答がなければ相手が故障した」と確実に判定できる | 合意(consensus)は、故障数が一定以下(過半数等)なら、決定的に到達可能。理論上最も強い保証 |
| **部分同期(Partially Synchronous)** | 遅延・処理速度に上限は存在するが未知、または「ある時点(GST: Global Stabilization Time)以降にのみ」上限が成立する | 安全性(誤った合意をしない)は常に保証できるが、到達可能性(必ず合意に至る)はGST以降にのみ保証される。PBFT・Raft等、実務の合意プロトコルの多くがこの前提に立つ |
| **非同期(Asynchronous)** | メッセージ遅延・処理速度に一切の上限がない | **FLP不可能性定理**(Fischer, Lynch, Paterson, 1985): 1つでも故障しうるプロセスがあれば、完全非同期環境では決定的な合意プロトコルは存在しない(安全性と到達可能性を両立できない)。到達可能性は原理的に保証できない |

### 1.2 SMASにとっての含意: 通信モデルは「戦略的な逸脱の温床」にもなりうる

分散システム理論では通信モデルは主に「故障(crash fault)への耐性」の文脈で語られるが、SMASの文脈では、**通信モデルの選択が、②誘因構造層の前提そのもの(手続き的P2P性)を、ネットワークレベルで静かに壊しうる**、という追加の含意がある。

**具体的なリスク**: 非同期モデルでは、戦略的なエージェントが**自分の申告を意図的に遅らせ、他のエージェントの申告を先に観測してから、自分の行動を決める**ことができてしまう。これは、`architecture_family_map.md` 2.3節で確認した**「優先権の非対称(Stackelberg型)」を、プロトコル設計上は排除したつもりでも、ネットワークの非同期性を通じて事実上復活させてしまう**、という抜け道になる。

**含意**: SMAS本体(手続き的P2P性=全員が同時に、同じルールに服する、という前提)を額面通り実現したいなら、**①環境層の通信モデルは、少なくとも部分同期以上(理想的には同期に近い、"申告の受付を一定時間で締め切る"というラウンド制)を採用する必要がある**。完全非同期モデルは、FLP不可能性による到達可能性の理論的な弱さに加え、SMAS固有の「優先権非対称の忍び込み」という、二重のリスクを持つ。

### 1.3 1ケース目における推奨

1ケース目(タスク配分)は少数主体・単一プロセス内のシミュレーションのため、この問題は表面化しない(擬似的に同期モデルが成立する)。**複数プロセス・実ネットワークをまたぐ実装に移行する際に、この節の内容を再検討する必要がある**、という申し送りとして記録する。

### 1.4 具体的設計原則: フェーズ分離(通信遅延と戦略的逸脱の区別不能性への対応)

**問題**: 非同期環境では、「通信遅延で応答が遅い」ことと「相手が戦略的に情報を隠蔽し行動を遅らせている」ことが、外部から区別できない。この区別不能性は、1.2節で述べた「優先権の非対称の忍び込み」をより具体的に言い換えたものであり、放置すると③集約層に不確定な入力が流れ込む。

**対応原則**: ③集約層(メカニズム実行層)が計算を行う前に、**「合意形成のバッファ層」を明示的なフェーズとして設ける**。

```
[非同期フェーズ] 各エージェントからの申告・状態を収集する
        ↓ CRDT・ベクタークロック等で、
          入力状態の順序と収束をネットワークレベルで確定させる
[同期フェーズ] 確定したスナップショットに対してのみ、
          メカニズムデザイン(VCG等)を適用する
```

この「非同期な収集フェーズ」と「同期的な集約フェーズ」を明確に分離することで、①環境層(スナップショット確定)と③集約層(確定後のみ計算)の責務が切り分けられ、通信遅延が誘因構造の計算そのものに紛れ込むことを防ぐ。1ケース目(単一プロセス内)では両フェーズが自明に一致するため意識不要だが、複数プロセス化する際の設計原則として記録する。

### 1.5 検証力の序列(言語選定・優先度判断の土台)

これまでの一連の検討を通じ、検証層内での実質的な検証力の序列が明確になった。

- **③頑健性(誘因整合性・耐戦略性)を実質的に保証しているのは、ゲーム理論の証明・`pygambit`・モンテカルロ**
- **②収束性を実質的に保証しているのは、MDP(または将来の確率的ドメイン理論)**
- **圏論(構造検証)が保証しているのは、①〜⑤層の"型の接続"という、最も限定的な範囲**。事後チェックであり、絶対的な証明ではなくツールへの相対的な信頼(2.1節参照)。合成可能性が強く効くのは委任連鎖のみで、1回性エンジン(VCG)にはほぼ効かない

この序列は、技術選定(言語選定を含む)における優先順位判断の土台になる。圏論表現の美しさを理由に技術選定を左右すべきではない、という判断根拠として`CLAUDE.md`の言語選定結論と対応する。

### 2.1 MDPの適用限界と、確率的ドメイン理論による補完(発展的検討)

**MDPが前提とする構造**: 「状態(state)が存在し、行動によってある状態から別の状態へ遷移する」という、明確な状態遷移構造。信用枠配分のような繰り返し・時間発展ケースでは自然に成立するが、**1回性エンジン(VCG等)には"遷移する状態"自体が存在しない**ため、MDPの前提が原理的に成立しない場面がある。

**1回性エンジンでの現状の代替**: モンテカルロ(様々な入力パターンで逸脱が得にならないかを統計的に確認)、Gambit(離散的なら厳密解)で対応してきた(`smas_implementation_spec_for_cursor.md` 9章参照)。これは"収束確率"の代替ではなく、"耐戦略性の経験的頑健性"という、別の性質の検証への役割の振り替えだった。

**確率的ドメイン理論(Probabilistic Domain Theory)による、より一般的な補完(将来の拡張候補)**: MDPが"きれいな有限の状態遷移"を前提にするのに対し、確率的ドメイン理論は、プログラミング言語の意味論(denotational semantics)で再帰計算の収束を説明するために使われてきた、より抽象的な数学的枠組み。不動点定理により、"具体的な状態機械の形"を前提にしなくても、「ある操作を繰り返し適用すれば、必ず収束する」ことを、シミュレーションなしに証明できる可能性がある。**MDPが原理的に適用できない(状態遷移構造を持たない)局面での、収束性(②)の理論的な保証手段として、位置づけを再検討する価値がある**。

**現状の優先度**: `required_theory_fields.md`では「MDPによる実証評価で、アーキテクトとしての説得力には十分」として拡張候補(優先度低)に位置づけていたが、これは"より厳密にしたいから"という動機を想定したものだった。今回の検討で、"MDPが原理的に適用できない場面がある"という、"できない"という制約への対応という、より強い必要性が見えてきた。1ケース目(1回性のVCG)でモンテカルロ・Gambitによる代替が機能する限り優先度を上げる必要はないが、**将来、状態遷移を持たないケースで"収束性(②)そのもの"を厳密に語りたくなった場合、最初に検討すべき拡張候補として記録しておく**。

## 4. 束論とゲーム理論の接合(新規検討、OR-Set拡張の具体例)

### 4.1 束(semilattice)の基礎

**結合半束(join-semilattice)**: 任意の2元に対して、最小上界(join, ∨演算)が常に存在する半順序集合。この演算が**可換・結合的・冪等**であれば、**「どんな順序でデータを合成しても、最終的に同じ結果に収束する」ことが数学的に保証される**。これがCRDT(Conflict-free Replicated Data Type)の収束保証の基盤。

**OR-Set(Observed-Remove Set)**: 集合に対するCRDTの代表例。各要素の追加(add)に一意なIDを付与し、削除(remove)は「観測済みのadd」に対してのみ有効な"墓標(tombstone)"として記録する。これにより、**追加と削除が同時多発しても、順序に依存せず、正しく収束する**(削除されたはずの要素が復活する、追加されたはずの要素が消える、といった不整合が起きない)。

### 4.2 SMASへの接合(この会話における提案)

**基本認識**: 束論・CRDTが保証するのは「①環境層のデータ構造が、通信順序に関わらず正しく収束する」という**構造的な収束性(②の一部)**。一方、ゲーム理論(②誘因構造層)が保証するのは「エージェントが正直に振る舞うことが最適である」という**戦略的な正しさ(③頑健性)**。**この2つは独立した保証であり、片方が満たされても、もう片方は自動的には満たされない。**

**OR-Setの拡張案(仮説)**: SMASの①環境層で、痕跡(申告・実績)をOR-Set的な構造で管理する場合、通常のOR-Setは「追加・削除」という操作にのみ意味を持たせるが、SMASでは**「追加(add)に、支払い・評価に関わる値を持たせ、削除(remove、撤回)に、コストリー・シグナリングの考え方(前回確認した"撤回コスト")を組み込む」**という拡張が考えられる。

```
通常のOR-Set: add(要素, ID) / remove(要素, 観測済みID)
  → 構造的な収束のみを保証

SMAS拡張版(仮称: Incentive-Weighted OR-Set):
  add(申告, ID, 値) / remove(申告, 観測済みID, 撤回コスト)
  → 構造的な収束(束論が保証)と、
    戦略的な正しさ(撤回が安易に得にならないよう、
    ②誘因構造層のルールで撤回コストを設計する)を、
    同じデータ構造の中で明示的に結びつける
```

**含意**: この拡張が機能すれば、**「①環境層のデータ構造が収束する」ことと「②誘因構造層が耐戦略性を満たす」ことを、別々に検証するのではなく、1つの拡張されたCRDT構造の中で、両方を同時に扱える**可能性がある。ただし、これは**まだ検証されていない、この会話内での思考実験レベルの提案**であり、実際にOR-Setの数学的性質(可換性・結合性・冪等性)を、支払い・撤回コストという要素を加えても保てるかは、個別の証明が必要。

## 5. 誘因構造層の技術選定における未決事項(既存の記録の集約)

`tech_selection_candidates.md`で触れた未決事項を正式に記録する。

この選定は、④資源コストという大指標の測り方そのものを左右するため、Ph2-7(サブタイプ分類)の結果と連動して決めるべき、未決事項として記録する。

| **偽名入札耐性(false-name-proofness)**(横尾ら) | VCGが偽名入札に脆弱であることの証明と、耐偽名メカニズムの設計 | SMASは参加者の同一性を明示的な前提として宣言し、解決しない(`scope_exclusions_and_deferrals.md` Part 0)。オープンなAIエージェント環境への適用を語る際は、この前提の強さを必ず明示する |
| **環境の第一級抽象化(MAS研究)**(Weyns らの environment abstraction、Agents & Artifacts モデル) | ①環境層(痕跡ストア・スティグマジー・環境を介した間接調整)の発想の直接の先行 | 「環境を介した間接調整」自体は確立された概念であり、①環境層を説明する際の引用元候補 |
| **電子制度・規範的MAS**(electronic institutions: ISLANDER/AMELI 等) | ルールの静的配置・アクセス制御・「制度」としてのメカニズムという枠組みの先行 | 「制度・ガバナンス」を明示区画とするアーキテクチャの先行。骨格層再構成案(保留中、D-13)の検討時に参照する |
