# 評価観点 一覧 (Evaluation Criteria)

## ①概念層由来の評価観点(なぜ良い設計か) — Conceptual Layer

1. **誘因整合性(Incentive Compatibility)** — 正直な行動が最適になっているか
2. **耐戦略性(Strategy-Proofness)** — 嘘・操作で得をする余地がないか(注: 耐戦略性は誘因整合性の最強形(支配戦略誘因整合性)であり、1と2は独立した別性質ではなく包含関係にある。1=広義の達成、2=その最強水準として測る。`DECISIONS.md` D-09)
3. **情報の非対称性の制御(Information Asymmetry Control)** — 優位な情報を持つ主体による搾取が起きないか
4. **必要多様性(Law of Requisite Variety)** — 制御側が対象の複雑さに見合う多様性を持つか

## ②誘因構造層由来の評価観点(何を設計したか) — Incentive Structure Layer

5. **結託耐性(Collusion Resistance)** — 複数主体が結託しても誘因構造を出し抜けないか
6. **コストリー・シグナリングの妥当性(Costly Signaling Validity)** — 偽装コストが偽装の利得を上回っているか
7. **時間発展・繰り返しゲームでの安定性(Stability under Repeated Games)** — 割引率・評判の蓄積が長期的に機能するか

## ③集約層由来の評価観点(結論の出し方は妥当か) — Aggregation Layer

8. **決定論性・局所-大域整合性(Determinism / Local-Global Consistency)** — 各主体がローカル計算しても同じ結論に至るか
9. **集約の操作耐性(Aggregation Manipulation Resistance)** — 集約ルール自体が戦略的に歪められないか
10. **公平性(Fairness / Envy-freeness)** — 資源配分・結果が特定主体に不当に偏らないか

## ④実行主体層由来の評価観点(主体の実装に依存しないか) — Agent Layer

11. **プラガブル性(Pluggability / Implementation Independence)** — ルールベース/最適化/POMDP/LLMのどれでも同じ誘因構造が機能するか
12. **適応的逸脱への頑健性(Robustness to Adaptive Deviation)** — 検出ルールを学習して回避するような、巧妙な逸脱にどこまで耐えるか

## ⑤検証層由来の評価観点(構造として矛盾ないか) — Verification Layer

13. **合成則の充足(Composition Law Satisfaction: Associativity, Unitality)** — 部品を組み合わせても構造が壊れないか
14. **権力集中の不在(Absence of Concentrated Power)**(旧: 単一障害点の不在) — 「止めたら全体が止まる」というインフラ的な壊れやすさではなく、**特定の1主体が他主体の運命を一方的に決める権限を持っていないか**という観点。規模を問わず意味を持ち、少数主体(2〜5体)の系でも成立する。**介入ポート(ルールのオフライン更新権限)・ラウンド締め切りの権威・支払いの執行主体も、この観点の監視対象に含める**(中央性の残余、`scope_exclusions_and_deferrals.md` Part 0)
15. **新規ケースへの対応づけ可能性(Functorial Transferability)** — 別ドメインに構造を保ったまま移植できるか

## 実行時・実証的な評価観点(評価手法の確立) — Empirical Verification

16. **並行安全性(Concurrency Safety)**(競合状態・デッドロックの不在) — Quint(TLA)による形式検証
17. **収束性(Convergence)**(繰り返しの結果、安定状態に到達する確率・速度) — マルコフ決定過程による定量評価
18. **通信密度の臨界点(Percolation Threshold)** — 痕跡の密度が上がった際にカオス(機能不全)に転じる閾値(優先度低・将来拡張)

## 横断的な評価観点 — Cross-Layer Criteria

19. **委任連鎖を通した誘因構造の伝播の妥当性(Incentive Propagation through Delegation Chains)** — 再委任のたびに誘因構造が継承・劣化しないか
20. **逸脱注入からの回復力(Recovery from Deviation Injection)** — 意図的な逸脱を入れた際、系が環境側の作用で自己修正するか

## 精緻化項目(既存の柱の下位項目として追加) — Refinement Items

21. **参加者の入退場に対する頑健性(Robustness to Population Dynamics)** — 新しい主体が参加/離脱しても、誘因整合性・集約の性質が崩れないか(①②の精緻化)
22. **開示情報の最小性(Disclosure Minimality)** — 協調を成立させるために、主体はどれだけの情報を晒す必要があるか。晒す量が少ないほど良い設計(①情報の非対称性の制御、の精緻化)
23. **打ち切り耐性(Termination Robustness)** — 最大試行回数・タイムアウトによる強制打ち切りが、①耐戦略性を壊す新たな逸脱の温床(意図的に収束を遅らせて打ち切りを誘発する等)にならないか(②③の精緻化)
24. **資源コスト(Resource Cost)** — 誘因構造エンジンの計算コスト(分散台帳のガス代、検証可能遅延関数の計算量等)、通信コスト。大指標体系(`roadmap_consistency_memo.md`参照)の④資源コスト系に対応する、新規追加項目。サブタイプ選定時に、経済的資源型(スマートコントラクト等)と計算コスト型(PoW系等)のどちらの誘因構造エンジンを採用するかを左右する

## 網羅性についての評価

**十分と考えて良いです。** 理由:

- 1〜4は①概念層(なぜ)、5〜7は②誘因構造(何を)、8〜10は③集約(どう結論を出すか)、11〜12は④実行主体(実装非依存性)、13〜15は⑤検証(構造として妥当か)、16〜18は実証(本当に動くか)、19〜20は層をまたぐ統合的な観点、というように**5層+評価手法という中核要素と1対1で対応しており、抜け漏れなく層構造にマッピングできる**
- 新しい観点が今後見つかったとしても、**基本的には上記のどれかの精緻化(例: 「結託耐性」をより厳密にする、「収束性」の証明範囲を広げる)として位置づけられる可能性が高く、新しい層や新しい柱を必要とする性質のものが出てくる可能性は低い**
- ただし、**「想定していないドメインに適用したときに初めて気づく評価観点」が出てくる可能性は残る**(これは以前合意した通り、2ケース目以降の実装で検証すべき領域)。網羅性は「現時点で構想できる範囲では十分」であり、「未来永劫これで完結する」という意味ではない
