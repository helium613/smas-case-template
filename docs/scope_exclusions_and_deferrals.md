# スコープ外・拡張フェーズ回し 項目一覧(理由付き)

## Part 0: 明示的な前提(解決せず"仮定する"と宣言する項目)(`DECISIONS.md` D-10)

Part 1(スコープ外)・Part 2(拡張保留)と同じ流儀を、これまで暗黙だった前提に適用する。以下は「解決すべき課題」ではなく、**本アーキテクチャが外生的に保証されると仮定する条件**。仮定が崩れる環境では①概念層の性質保証は成立しない。対外的には「完全な非中央集権」ではなく「中央機能の最小化と、残余の明示」として説明する。

| 前提 | 内容 | 仮定が崩れた場合 |
|---|---|---|
| **参加者の合理性** | メカニズムの性質保証(IC/SP/IR)は、合理的効用最大化主体に対する保証。LLM等の非合理的逸脱は理論保証の対象外であり、モンテカルロ・Red Team Agentで経験的に確認するのみ | 耐戦略性は「合理的逸脱が得にならない」ことしか守らず、非合理的逸脱が効率性・他者に与える損害は防げない |
| **参加者の同一性(Sybil耐性)** | 偽名申告・複数アイデンティティの排除は外生的に保証されると仮定。VCGは偽名入札(false-name bidding)に脆弱であることが知られている(横尾ら) | オープンな(参加障壁のない)AIエージェント環境では、偽名入札により耐戦略性が崩れる。同一性検証の仕組みはスコープ外 |
| **支払いの執行と沈め先** | VCG支払いの徴収・執行、および支払いの受け手(参加者へ再分配すると誘因が壊れるため外部の沈め先が必要)の存在を仮定。VCGは予算均衡を満たさない(Green–Laffont) | 執行機構が存在しなければ支払いルールは空文化する。執行機構それ自体が中央性の残余 |
| **ラウンド進行の権威** | 同期ラウンド(申告の一斉締め切り)を宣言する時計・進行役の存在を仮定(`SMAS_theorymap.md` 1章の同期前提の帰結) | 締め切りの権威なしでは、非同期性を通じて優先権の非対称(後出し)が復活する |
| **効用の準線形性** | VCGの適用条件(効用=価値−支払い、という形)を仮定 | 準線形でない効用(予算制約・リスク回避等)ではVCGの耐戦略性は保証されない |
| **壁の執行力の限界** | Pydanticバリデーションはプロセス内の型検証であり、悪意あるコードの迂回を防ぐ執行機構ではない。1ケース目(単一プロセス)では十分と割り切る | 複数プロセス・実ネットワーク化の際、アクセス制御の執行方式を再設計する必要がある |
| **介入ポートの統治** | ルールをオフライン更新できる主体の存在を設計上認めている。これは統治権力であり、評価観点14番(権力集中の不在)の監視対象に含める | 介入ポートの更新権限が特定主体に集中したまま無監視だと、「権力集中の不在」の主張と矛盾する |

## Part 1: 意図的にスコープ外とした項目(恒久的な非対象)

これらは「今回のリソース不足で後回し」ではなく、**SMASという対象系(非協力ゲーム理論ベース、戦略的主体前提)の定義そのものと、性質が異なるため対象外**とした項目。

| 項目 | スコープ外の理由 |
|---|---|
| **インフラの可用性・冗長化・耐障害性** | ノード故障・ネットワーク分断への対応は分散システム工学の問題であり、「誘因設計」とは異なる問題設定。SMASの差別化軸(戦略的に嘘をつく主体への対処)とは無関係 |
| **協力ゲーム理論(提携形成・シャープレイ値等)** | 「誰と誰が組むと得か」という問いは、SMASが前提とする非協力ゲーム理論の枠組みと数学的道具立てが根本的に異なる。②誘因構造層の中身が総取り替えになる、別トピック |
| **ビザンチン故障耐性(BFT)** | SMASの誘因構造は「自己利益のために合理的に嘘をつく主体」を前提とする。「自分の利得を犠牲にしてでも系を破壊したい主体(悪意・妨害目的)」には、誘因設計は原理的に無力 |
| **アクターネットワーク理論(ANT)** | 数学的検証ができない、社会学の解釈フレームワーク。厳密な概念層(数学)と解釈的な語り口(社会学)が混ざると、粒度の不揃いが再燃するため、アーキテクチャ構成要素にはしない |
| **真にボトムアップな創発的ガバナンス(Ostrom型)** | ①環境層の前提(設計者がルールを配置する)が根本から崩壊する。「誰も設計していないのに規範が自然に立ち現れる」という性質は、SMASのトップダウン設計の前提と非対称。5層構造もほぼ再設計が必要になる、別アーキテクチャの対象 |
| **通信プロトコルそのものの設計・A2Aを前提にすること** | SMASの差別化は「協調の作法(誘因構造)」であり「接続性そのもの」ではない。A2Aは信頼できる主体同士の相互運用性を主眼にしており、戦略性を前提にしていないため、依存すると汎用性が狭まる |
| **SMAS独自の通信プロトコル新規開発** | Agent OSI ModelのL1〜L4(実行・通信・発見・セッション)は既に標準化が進んでいる領域。土管の再発明は差別化にならず、時間の浪費になる |
| **隠れた行動(Moral Hazard)への対応** | SMASの現状の逸脱モデルは「申告段階での正直さ」(隠れた情報)にのみ焦点を当てている。「申告は正直だったが、事後の実行が手抜きだった」という逸脱を検出するには、①環境層が結果だけでなくプロセスも痕跡として観測できる必要があり、現状の痕跡設計(結果中心)とは別の設計軸を要求するため対象外。**実行品質の評価はドメインごとに異なる評価関数を要し、②誘因構造層と同様に"重い"要素になるため、①環境層をケースごとに重くしてしまい、SMASの軽量な骨格という設計思想と相容れない。最終的な位置づけとして、①環境層のA側(`environment_schema.py`)に`process_trace: Optional[dict] = None`という任意の拡張ポイントのみ用意し、中身の設計は個別ケースの実装者に委ねる、というアーキテクチャ制限として明示する。SMAS本体のコアスコープには含めない** |

## Part 2: 拡張フェーズ回し(将来検討する、優先度低・保留)

これらは「対象外」ではなく、**構造的には組み込み可能だが、1ケース目の疎通確認という現在のスコープでは過剰、または学習コスト・時間軸との兼ね合いで後回しにした**項目。

### 理論・検証手法

| 項目 | 保留理由 | 再検討のタイミング |
|---|---|---|
| **代数的位相幾何学(合意形成の不可能性証明)** | 学習コストが圏論以上に高く、現フェーズの投資対効果に見合わない | ⑤検証層をさらに厳密化したくなった時 |
| **計量セマンティクス・確率的ドメイン理論(不動点定理)** | MDPによる実証的な収束確率評価で、アーキテクトとしての説得力には十分。研究レベルの完全証明は過剰投資 | 研究レベルの厳密性が必要になった時 |
| **PRISM(確率的モデル検査)** | Quintと役割が一部重複(モデル検査系ツール2つ持つのは過剰)。Python外のツールで新規学習コストが発生 | 耐戦略性を数学的に完全証明したいレベルまで厳密化したくなった時 |
| **パーコレーション理論・グラフ動態** | 痕跡密度の臨界点分析。1ケース目の規模(少数主体)では、この現象自体が起きにくい | エージェント数が数百規模になるケースに着手する時 |
| **自由エネルギー原理・アクティブインファレンス** | ④実行主体層の一実装オプション(POMDPの一般化)に過ぎず、②誘因構造層を「予測エラーに画一化」すると①概念層の厳密性(ゲーム理論的な証明可能性)を失うリスクの方が大きい | POMDPの拡張として、より高度なエージェントを実装したくなった時 |
| **オートポイエーシス理論** | 新しい層やエンジンではなく、既存の評価観点(参加者の入退場・委任連鎖)を考える際の解釈のレンズに留まる | 評価観点21・委任連鎖の議論を深める時 |
| **自動メカニズムデザイン(メカニズムソルバー)** | 制約から支払い関数を自動探索・生成するのは研究レベルの難度。カタログ(既知メカニズムから選ぶ)で1ケース目は十分 | カタログにない制約に直面し、かつ研究的探究に時間を割ける段階になった時 |
| **構成的アプローチ(Correct-by-Construction)** | Pythonの言語的限界(動的型付け)により、完全な実現は不可能。検証的アプローチ(事後チェック)で1ケース目は十分機能する | 2ケース目着手時に最優先で検討(他の拡張候補より学習コストが低いため) |

### 実行主体層(④)

| 項目 | 保留理由 | 再検討のタイミング |
|---|---|---|
| **POMDP** | 部分観測の精緻なモデル化は1ケース目に不要。`pomdp-py`の学習コストもある | 情報の非対称性そのものを主題化したいケースに着手する時 |
| **最適化ベースエージェント** | ①誘因構造層への依存が強く、プラガブル性が部分的に弱まる実装。1ケース目はルールベース+LLMモックで基礎検証は十分 | 誘因構造の理論的厳密性を上げたい時(POMDPより優先度は高い) |
| **LLM実物+HTTPジッター再現(`respx`等)** | デモの演出リッチさのみに寄与し、理論検証には不要 | デモの完成度を上げたいタイミング(任意) |

### 評価観点(コア/拡張の区分)

| 項目 | 保留理由 |
|---|---|
| **結託耐性(完全な形式証明)** | 1ケース目は「想定する結託の範囲を明示する」程度の最低限の考慮で足り、完全な証明までは不要 |
| **公平性(Envy-freeness)** | ③集約層の中核ではあるが、1ケース目(VCG中心)では優先度は中程度に留める |
| **適応的逸脱への頑健性** | 検出ルールを学習して回避する巧妙な逸脱への対処。単純な逸脱注入で1ケース目の主張は成立するため後回し |
| **通信密度の臨界点** | 上記パーコレーション理論と同じ理由 |
| **参加者の入退場への頑健性** | ①②の精緻化項目。1ケース目は固定メンバーでの疎通確認が主眼のため後回し |
| **開示情報の最小性** | ①情報の非対称性の精緻化。1ケース目の基礎検証には必須でない |

### リポジトリ・運用

| 項目 | 保留理由 | 再検討のタイミング |
|---|---|---|
| **ケース間のインスタンス連携(smas-case-*同士の接続)** | 現状スコープ外として明示。1ケース目は単独で完結させる | 実際にタスク配分の結果を信用枠配分に連携させたくなった時(2ケース目以降) |
| **ライフサイクル管理(バージョニング・非推奨化)の正式な仕組み** | 外部利用がまだない段階では過剰。README注記程度で当面は十分 | 外部からのforkが実際に増えてきた時 |
| **③→②への昇格プロセスの明文化** | 「2ケース目実装後に③から②へ書き戻す」という手順自体は既に合意済みで、正式なルール文書化は不要 | 3ケース目以降、昇格の判断が頻発するようになった時 |
| **可視化プレゼン層の独立リポジトリ化** | ①環境層のログ出力フォーマットが標準化されるまでは、ケース内にベタ書きで十分 | 2ケース目に着手し、共通化できると気づいたタイミング |

## 横断的な傾向: 保留判断の共通基準

Part1(恒久的スコープ外)とPart2(拡張フェーズ)を分ける基準は一貫している。

- **Part1**: 「SMASが前提とする"戦略的・非協力・トップダウン設計"という定義そのものと、性質が違う」→ 組み込もうとすると①概念層・②骨格層の前提が壊れる
- **Part2**: 「性質としては組み込み可能だが、"アーキテクトとしての説得力で十分、研究者レベルの完全性は求めない"という目的水準に対して、今は投資対効果が低い」→ 1ケース目の疎通確認、または2ケース目以降の展開の中で、必要になったタイミングで拾い直す

この基準により、**「なぜやらないか」を都度個別に判断するのではなく、機械的に分類できる状態**になっている。
