# SMAS (Strategic Multi-Agent System) 実装仕様書
### Cursor等でのスクラッチ実装用インプット / v1.0

---

## 0. プロジェクト概要

**一言定義**: 非中央集権的協調が成立するための3性質(誘因整合性・耐戦略性・個人合理性)を、5層構造として設計し、層間の型接続の整合性を機械的に検証しつつ、実行時の安全性・収束性も実証する、アーキテクチャパターンとその検証方法論。(圏論の位置づけの降格は`DECISIONS.md` D-08)

**対象系**: 戦略的多主体協調系(Strategic Multi-Agent System, SMAS)
- 「戦略的」= 各主体が自己利益に基づき、嘘・逸脱をしうることを前提とする
- 中央の執行者(オーケストレーター)を置かず、環境を介した間接的な作用で協調と逸脱修正を実現する

**このドキュメントの目的**: 1ケース目の簡易スクラッチ実装(疎通確認レベル)を行い、①アーキテクチャの各層インターフェースが実際に噛み合うか、②評価観点が実用的に測定可能か、を検証する。

---

## 1. スコープ

### 対象とする
複数の自律的な主体が、限られた資源・情報を巡って、正直な行動が必要になる協調様式全般(資源配分、投票、委任・権限移譲、信用形成、市場マッチング等)

### 対象としない(意図的にスコープ外)
- インフラレベルの可用性・冗長化・耐障害性(ノード故障、ネットワーク分断、ビザンチン耐性)
- 独自の通信プロトコル設計(A2A等は「配線」として扱い、規格化しない。JSON Schemaレベルでの互換性のみ意識)
- LLMの内部推論品質そのもの
- 協力ゲーム理論(提携形成)、真にボトムアップな創発的ガバナンス(Ostrom型)は別アーキテクチャの対象

## 2. アーキテクチャ: 3階層 × 5層構造

### 2.1 3階層モデル

| 階層 | 内容 | 普遍性 |
|---|---|---|
| **①概念層** | 中核3性質(誘因整合性・耐戦略性・個人合理性)+前提背景(情報の非対称性)+メタ理論(必要多様性の法則、不完備契約理論) | 完全に普遍 |
| **②骨格層** | 5層構造(下記) | ほぼ普遍(2ケース目で検証予定) |
| **③具体設計層** | ケースごとの数式・パラメータ。ただし「メカニズムのカタログ」は再利用可能な中間層として保持 | ケース依存 |

### 2.2 5層構造(データフロー版、プレゼン・説明用)

```
①環境層(Environment) → ②誘因構造層(Incentive Structure) → ③集約層(Aggregation) → ④実行主体層(Agent) → ⑤検証層(Verification)
```

| 層 | 責務 | 実装の要点 |
|---|---|---|
| **①環境層** | 痕跡ストア、壁(アクセス制御)、ルールの静的配置、介入ポート(バージョン管理)、減衰関数 | dict/JSON永続化からスタート。Pydanticで壁を型定義。**`process_trace: Optional[dict] = None`という、実行品質(モラルハザード対応)向けの任意拡張ポイントをA側の型に含めておく(中身の設計はスコープ外、個別ケース実装者に委ねる)**。**痕跡の中身(`Trace.payload`)は、②誘因構造層と同じA/B構造(ジェネリック型)にする。`Trace[T]`のTをケースごとに定義し(例: `TaskAllocationPayload`, `CreditPayload`)、「何でも入るdict」にしないことで、①③間の型チェックを圏論検証で意味あるものにする** |
| **②誘因構造層** | 正直な行動が支配戦略になるよう設計。**誘因構造エンジンを内包(A側=規格、B側=個別実装)** | 1ケース目はVCGメカニズムを採用 |
| **③集約層** | 決定論的な集約関数、操作耐性、局所計算・大域整合 | 集約計算はファミリー依存(VCG系の配分決定=割当問題は`scipy`、投票系は`pref_voting`) |
| **④実行主体層** | プラガブルな意思決定ロジック(ルールベース/最適化/POMDP/LLM) | 優先度順: ルールベース→LLMモック→LLM実物→最適化→POMDP |
| **⑤検証層** | 複数の検証手段を束ねる層(詳細は`verification_layer_clarification.md`参照)。構造検証(圏論・DisCoPy、合成則・可換性)/動的安全性検証(Quint、並行性・打ち切り安全性)/実証的検証(モンテカルロ・Gambit、耐戦略性)/収束検証(MDP、収束確率)の4サブコンポーネント | 構造検証は**検証的アプローチ**(事後チェック)を採用。構成的アプローチは2ケース目以降の拡張候補。評価対象は各エンジンの「境界(層をまたぐ接続の整合性)」であり「中身(数式そのものの正しさ)」ではない |

### 2.3 責務分離版(実装のマスター、内部設計用)

データフロー版とは別に、以下の責務ベースの構造を実装時のマスターとする:

| 責務層 | 内容 |
|---|---|
| **ストレージ層** | 痕跡の読み書き・減衰のみを担う。ロジックは持たない受動的な層(旧①環境層の「保持」機能) |
| **メカニズム定義層** | 配分ルール+支払いルールを不可分な1つの仕様として定義(旧②誘因構造層+③集約層を統合) |
| **メカニズム実行層** | ストレージ層から読み取り、メカニズム定義層のルールを計算・適用する |
| **主体決定層** | 各エージェントの意思決定ロジック。メカニズム定義層を読み取り専用データとして参照してよいが、ロジックに埋め込んではいけない |
| **構造検証層** | 上記の組み合わせが性質を満たすか検証 |
| **(横断)アクセス制御** | 「誰が何を読み書きできるか」。独立層ではなく、ストレージへの全アクセスに適用される横断的制約 |

**方針**: コードのファイル分割は責務分離版に従う。データフロー版は説明・可視化用。

## 3. ①概念層: 詳細

### 中核3性質(定理レベルの厳密さ、既存学術定理を引用すればよく自前証明は不要)
1. **誘因整合性(Incentive Compatibility)**: 正直な行動が最適になっている
2. **耐戦略性(Strategy-Proofness)**: 嘘・操作で得をする余地がない
3. **個人合理性(Individual Rationality)**: 協調に参加することが、参加しないことより損にならない

**注(包含関係とトレードオフ、`DECISIONS.md` D-09)**:
- 耐戦略性は誘因整合性の最強形(支配戦略誘因整合性)であり、1と2は独立した別性質ではない。「1=広義の達成、2=その最強水準の達成」として扱う
- 効率性・誘因整合性・予算均衡・個人合理性の同時達成は一般に不可能(Myerson–Satterthwaite、Green–Laffont)。VCG採用は**予算均衡を諦める**選択であり、支払いの沈め先(徴収・執行の主体)の存在は明示的な前提として宣言する(`scope_exclusions_and_deferrals.md` Part 0)

### 前提背景
- **情報の非対称性**: 隠れた情報(Adverse Selection、①〜③層が主対応)/隠れた行動(Moral Hazard、現状スコープ外)

### メタ理論(比喩的援用、注記として扱う)
- 必要多様性の法則(Ashby) — 制御側が対象の複雑さに見合う多様性を持つべき、という設計原則
- 不完備契約理論 — 介入ポート(事後的なルール更新)の必要性を裏付ける

## 4. 誘因構造エンジンの設計(②の核)

### 2階建て構造
- **A側(アーキテクチャ側/schemas/)**: `IncentiveStructure<P>`型。`payment_function`, `parameters`, `version`を持つインターフェース。介入ポート(オフラインでのバージョン更新)を含む
- **B側(ケース実装側/engine/)**: 具体的な支払い関数(VCG等)。**②誘因構造層のみが、ケースごとに新規の数学的導出を要する「エンジン」であり、他の層は「機構」として使い回せる**

### メカニズムカタログ(B側の出発点、再利用可能な部品棚)

1ケース目実装時、以下から選定する:

| ファミリー | 代表メカニズム | 耐戦略性 | 優先度 |
|---|---|---|---|
| 1回性・移転可能効用あり | VCGメカニズム | 満たす | **1ケース目の主軸** |
| 1回性・移転可能効用あり(悪い例) | 第一価格オークション | 満たさない | 検証層のテストケースとして併用 |
| 1回性・支払いなし(投票) | 単純多数決/ボルダ得点 | 満たさない場合あり | 対比用、実装コスト低 |
| マッチング | Gale-Shapley | 提案側は満たす | 2ケース目候補 |
| 繰り返し・評判ベース | トリガー戦略 | フォーク定理下で満たす | 2ケース目候補(信用枠配分) |
| 委任連鎖 | Liquid Democracy | 循環委任防止等の固有課題あり | 圏論の合成可能性が最も強く効くため、将来の目玉ケース |

## 5. 技術スタック(Python一本化)

| 層 | ライブラリ |
|---|---|
| 環境層 | dict/JSON永続化(将来Redis)、Pydantic(壁の型定義) |
| 誘因構造層 | 自作(支払い関数)、検算にGambit(`pygambit`) |
| 集約層 | ケース1(VCG)の配分決定は`scipy`の割当解。`pref_voting`は投票ファミリー用。収束確率評価に`pymdptoolbox` |
| 実行主体層 | ルールベース自作→将来`PettingZoo`/`pomdp-py` |
| 検証層 | `DisCoPy`(モノイダル圏・合成則) |
| 検証キット(独立併走) | Quint(TLAモード、並行安全性)、Python(MDP収束確率) |
| エージェントフレームワーク | **LangGraph**(1ケース目で基盤実装まで実施。適合度検証が目的の一つ) |

### 実行主体層の実装優先度
1. ルールベース(基準点、必須)
2. LLMモック(確率分布のみ、モンテカルロ用、必須)
3. LLM実物(Anthropic API、Tool Use/JSON Schema、目玉シーンのみ数回)
4. 最適化ベース(拡張、②への依存が強いため注意)
5. POMDP(拡張)

## 6. LLMエージェントのインターフェース設計

- **Tool Use/JSON Schemaで構造化出力を強制する**(自然言語のみのやり取りは禁止)
- 出力例:
```json
{
  "action": "bid | abstain",
  "declared_value": "number",
  "reasoning": "string (人間可読用、システムロジックはこれに依存しない)"
}
```
- LLMモックは、この同じJSON Schemaを返す、確率分布ベースの軽量関数として実装する(HTTPレベルのジッター再現は演出用途のみ、検証には不要)

## 7. LangGraphフレームワーク統合ガイドライン

- **既存フレームワークは④実行主体層の一実装オプションとして扱い、SMAS独自のフレームワークは新規開発しない**
- **状態オブジェクトの薄いプロキシ化ルール**: LangGraphのStateは、①環境層への参照(EnvironmentClient)のみを持たせ、データの実体を直接持たせない
- **統合時の3項目チェックリスト**:
  1. 状態の実体は①環境層にあるか(Stateは参照のみを持つか)
  2. 壁(アクセス制御)はフレームワークを迂回されていないか
  3. フレームワークのオーケストレーション(ノード実行順序)が②③⑤の役割に越境していないか

```python
# NG
class State(TypedDict):
    agent_data: dict  # 直接データを持たせない

# OK
class State(TypedDict):
    environment_ref: EnvironmentClient  # ①環境層への参照のみ
```

## 8. 具体設計レベルの必須要件(打ち切りルール)

③集約層(メカニズム実行層)に以下を実装する:
- **最大試行回数(Max Iteration)**、**タイムアウト(Wall-clock Timeout)**、**打ち切り時のフォールバック挙動**を定義する
- **打ち切りルール自体が、①耐戦略性を壊す新たな逸脱の温床(意図的に収束を遅らせて打ち切りを誘発する等)にならないよう設計する**
- 打ち切りイベントは、Quintでの並行安全性検証・モンテカルロでの耐戦略性検証、両方の対象に含める

## 9. 評価観点(1ケース目で確認すべきコア項目)

以下7項目を、今回のスクラッチで実際に測定できる状態にする:

1. 誘因整合性(①)
2. 耐戦略性(①)
3. 決定論性・局所-大域整合(③)
4. プラガブル性(④、ただし②への依存の強さに注意)
5. 合成則の充足(⑤)
6. 権力集中の不在(⑤、旧SPOF。「特定の1主体が他主体の運命を一方的に決める権限を持たないか」。少数主体でも意味を持つ)
7. 逸脱注入からの回復力(横断)
8. 打ち切り耐性(横断、追加)

**拡張項目(2ケース目以降)**: 結託耐性、公平性、適応的逸脱への頑健性、通信密度の臨界点、参加者の入退場への頑健性、開示情報の最小性

### 評価手法の使い分け(1回性エンジンの場合)
- **並行安全性**: Quint(TLAモード)
- **耐戦略性の経験的頑健性**: モンテカルロ(1回性エンジンでは「収束確率」ではなく「様々な入力パターンで逸脱が得にならないか」を検証する役割)。可能ならGambitで厳密な均衡計算も併用
- **収束確率**: `pymdptoolbox`(1回性エンジンでは適用対象外、繰り返し・時間発展ケースで使用)

## 10. デモシナリオ(3シーン構成)

```
シーン1(平常時): 正直申告 → 集約 → 配分決定
シーン2(逸脱注入): 1エージェントが過大申告(または他の定義済み逸脱パターン)
シーン3(自己拘束の確認): 同一ラウンド内で、逸脱エージェントの実現効用を
  正直申告時の効用(反実仮想)と比較し、「逸脱しても得をしない」ことを
  数値で示す(耐戦略性の直接実証)
```

**旧シーン3(痕跡→信用低下→取引忌避)は2ケース目(繰り返しゲーム)へ移設**(`DECISIONS.md` D-07)。理由: (a)1回性VCGでは逸脱は同一ラウンド内で自己懲罰的で、評判による事後修正は理論上不要、(b)逸脱の「検出」は観測不可能な真の評価額を要し、モラルハザード除外(スコープ外)と矛盾する。

**実装上の確認事項**: シーン3の反実仮想比較は計測・ログ出力として実装し、メカニズムの構成要素(ジャッジ)にしない。隠れた中央集権が紛れ込んでいないかの目視確認は全シーンで行う。

## 12. 今回のスクラッチのゴール(疎通確認レベル)

1. **1ケース目のドメインを確定する**(タスク配分を推奨。VCGで耐戦略性の基礎を固めやすく、実装コストが低いため)
2. 上記2〜11章の最小構成で、実際にコードを書いて動かす
3. コア評価観点8項目(9章)が、この実装で実際に確認できるかをチェックする
4. 「①②③概念層・骨格層は崩れなかったか」「5層という切り方に無理はなかったか」を記録する
5. LangGraphの状態プロキシ化ルールが、実際に無理なく実装できるかを検証する
6. ここでの気づきを、次のイテレーション(2ケース目: 委任・信用枠配分、または圏論の構成的アプローチへの拡張)の材料とする

## 13. 実装時の注意事項(これまでの議論からの申し送り)

- **②誘因構造層が最も"重い"(都度の数学的導出が必要)。他の層は極力共通実装を使い回す**
- **④実行主体層の「プラガブル性」は、"型の互換性"のみを保証し、"振る舞いの同等性"は保証しない**、という前提で設計する(LLMは理論通りに動くとは限らない)
- **⑤検証層は"絶対的な証明"ではなく"確立されたツール(DisCoPy)を介した相対的な安心材料"である**、という限界を自覚する
- **A2A等の外部プロトコルは前提とせず、④の入出力インターフェースだけJSON Schemaレベルで互換性を意識する程度に留める**
- **「アーキテクチャの空論化」への警戒**: 社会実装が進むほど、ドメイン固有実装(B側)が量的にA側を圧倒するのは当然の帰結であり、それ自体は問題ではない。**見極めるべき指標は「B側が増えるたびに、A側(schemas/)の書き換えがどれだけ発生するか」**。A側がほとんど変わらないままB側だけが厚みを増していけば健全、頻繁にA側を書き換える必要が生じたら設計に問題がある危険信号。痕跡の中身(`Trace.payload`)を含め、①〜⑤全ての層でA/B構造(型は共通、中身はケース依存)を徹底することが、この空論化を防ぐ一番の対抗策
