# ④実行主体層: 実装パターンの優先度

| 優先度 | パターン | 理由 | 実装コスト目安 |
|---|---|---|---|
| **1(最優先・必須)** | **ルールベース** | 5層構造・誘因整合性・逸脱注入テストという「主張の骨格」を最初に検証するための基準点。予測可能で、①〜⑤層のバグと"エージェントの気まぐれ"を切り分けられる。**これがないと他の何も検証できない** | 低 |
| **2** | **LLMモック(確率分布のみ、HTTPモックなし)** | モンテカルロでの理論検証に必要。ルールベースとほぼ同じ実装コストで、「戦略的に逸脱しうる主体」という前提をより自然に表現できる(固定ルールより確率的逸脱の方がメカニズムデザインの検証に適する) | 低 |
| **3** | **LLM実物(Anthropic API、目玉シーンのみ)** | デモの説得力・「本物で動く」という証明のために必要。ただし数回〜十数回の呼び出しに限定し、大量試行には使わない | 中(API結線のみなので実装は軽いが、プロンプト設計にやや時間がかかる) |
| **4** | **最適化ベース** | 「合理的な最適化主体」という、ルールベースとLLMの中間の意思決定モデル。①誘因構造の理論的性質(支配戦略など)を、より厳密な形で確認したい場合に効く。1ケース目必須ではない | 中 |
| **5(後回し可)** | **POMDP** | 部分観測という精緻さを加える拡張。理論的には魅力的だが、`pomdp-py`の学習コストもあり、1ケース目のデモ・検証には不要。2ケース目以降、あるいは「情報の非対称性」を主題化したい時に着手 | 高 |
| **6(演出用・任意)** | **LLM実物+HTTPモック(ジッター再現、`respx`等)** | デモの"リアルな体験"を演出したい場合のみ。理論検証にも1ケース目の説得力にも必須ではなく、時間があれば足す程度 | 低〜中(ライブラリ自体は軽いが、優先度としては最後) |

---

## 実装順序の推奨

```
1ケース目のデモに必要な最小構成:
  ① ルールベース(基準点)
  ② LLMモック(確率分布、大量試行用)
  ③ LLM実物(目玉シーン、数回のみ)

  → この3つだけで「主張の実証+デモの説得力」は成立する

2ケース目以降・精緻化フェーズで検討:
  ④ 最適化ベース(理論的厳密性を上げたい時)
  ⑤ POMDP(情報の非対称性を主題化したい時)
  ⑥ HTTPレベルのジッター再現(演出をリッチにしたい時)
```

## 判断基準の要約

- **①②③だけで「動く・検証できる・見せられる」の3条件は満たせる**ため、④⑤⑥は無理に1ケース目に詰め込まない
- ④⑤は理論の厳密性、⑥は演出のリッチさという、それぞれ異なる軸の追加価値であり、いずれも「なくても主張は成立する」拡張という位置づけ
