# LLMモック設計ガイド
### ポータブルスキルとして(SMASに限らず、LLMエージェントを含むシステムの検証全般に使える)

---

## 1. なぜLLMモックが必要か

LLMエージェントを含むマルチエージェントシステムを検証する際、以下2つの理由から「本物のLLM呼び出し」だけでは成立しない。

1. **コスト・時間の問題**: モンテカルロ的な大量試行(数千〜数万回)を、実際のAPI呼び出しで行うのは非現実的(課金・レイテンシ)
2. **検証の純度の問題**: LLMの言語的判断は複雑でノイズを含む。「①〜⑤層の性質(誘因整合性・収束性等)」を検証したいのに、「LLM特有の気まぐれ」が混ざると、何を検証しているか分からなくなる。モックの方がむしろ検証の純度が上がる

## 2. モックの2階層(用途別に作り分ける)

| 種類 | 何をモックするか | 用途 |
|---|---|---|
| **確率分布モック(推奨・基本形)** | 「意思決定の確率分布」のみ。レイテンシ・APIの挙動は再現しない | モンテカルロでの理論検証(大量試行) |
| **HTTPレベルモック(演出用・オプション)** | レイテンシ・エラー率・トークン単価等、APIらしい"挙動"を再現(`respx`, `vcr.py`等) | デモの"本物らしさ"を演出したい場合のみ |

**原則**: 理論検証には確率分布モックで十分。HTTPレベルモックは検証の目的には過剰で、デモ演出という別の目的にのみ使う。

## 3. 確率分布モックの実装方針

### 3.1 満たすべきインターフェース

本物のLLM呼び出しと**完全に同じ入出力インターフェース(JSON Schema)**を満たすこと。これにより、④実行主体層の「プラガブル性」(型として差し替え可能)が実際に機能する。

```python
# 本物のLLM実装
def decide_with_llm(observation: ObservationInput) -> ActionOutput:
    response = anthropic_client.messages.create(...)  # Tool Use
    return parse_to_action_output(response)

# 確率分布モック(同じシグネチャ、同じ戻り値の型)
def decide_with_llm_mock(observation: ObservationInput, p_honest: float = 0.8) -> ActionOutput:
    if random.random() < p_honest:
        return ActionOutput(action="bid", declared_value=observation.true_value, reasoning=None)
    else:
        deviated_value = observation.true_value * random.uniform(1.1, 1.5)  # 過大申告のシミュレーション
        return ActionOutput(action="bid", declared_value=deviated_value, reasoning=None)
```

### 3.2 パラメータ設計: 「何をモックするか」の最小要件

確率分布モックが持つべき最小限のパラメータは以下の2つ。

- **p_honest(正直に振る舞う確率)**: エージェントが理論通りの最適行動(多くの場合「正直な申告」)を取る確率
- **deviation_distribution(逸脱時の振る舞いの分布)**: 逸脱する場合、どの程度・どの方向に逸脱するか(過大申告、過小申告、ランダムなど)

この2つがあれば、「様々な入力パターンで、逸脱が得にならないか」というモンテカルロ検証(耐戦略性の経験的頑健性)に必要な最小構成が揃う。

### 3.3 モックの"忠実性"についての誠実な限界

- **モックの確率分布が、本物のLLMの振る舞いをどれだけ忠実に近似しているかは、検証されていない**。これは意図的な簡略化であり、「モックは"仮説上の確率分布"であり、実物と一致する保証はない」と明示しておく
- 将来、実LLMとモックの結果を比較検証する回を設ける、という拡張候補として保留してよい(1ケース目では過剰)

## 4. HTTPレベルモック(演出用)の位置づけ

- `respx`, `vcr.py`等のライブラリで、Anthropic SDKへのリクエストをインターセプトし、任意の遅延・任意のレスポンスを返せる
- **デモの"体験のリアルさ"には寄与するが、①②③の理論的性質(誘因整合性・収束性)の検証には本質的に無関係**
- 優先度は低く、時間があれば追加する程度でよい

## 5. モック運用のチェックリスト(汎用・ポータブル)

LLMエージェントを含む任意のシステムを検証する際、以下を確認する。

- [ ] モックは、本物と完全に同じ入出力の型(JSON Schema)を満たしているか
- [ ] モックのパラメータ(逸脱確率・逸脱の分布)は、検証したい仮説に応じて調整可能か
- [ ] 大量試行(モンテカルロ)には確率分布モックのみを使い、HTTPレベルモックは使っていないか(コスト・実行速度の観点)
- [ ] 「モックは実物の近似に過ぎない」という限界を、検証結果を報告する際に明示しているか
- [ ] デモの目玉シーンでは、本物のLLM呼び出しに切り替え、"モックではなく実際に動く"ことを示しているか

## 6. このガイドが汎用的である理由(ポータブルスキルとしての価値)

このモック設計の考え方(①型を揃えて差し替え可能にする、②確率分布とHTTP挙動を分離する、③大量試行にはコストの低いモックを使う、④モックの忠実性の限界を明示する)は、**SMASに限らず、LLMエージェントを含む任意のシステムのテスト・シミュレーション・CI設計に転用できる**。特に以下のような場面で応用できる:

- マルチエージェントのワークフロー(LangGraph等)のユニットテスト
- LLMを含むプロダクトのA/Bテスト設計(大量試行が必要な場面)
- LLM呼び出しのコスト見積もりのための、事前シミュレーション
