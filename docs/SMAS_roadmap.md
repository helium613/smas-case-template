# SMAS構想:全体ロードマップ(v2)

## 0. 全体構造:4フェーズ

「実装」と「検証・反復」は性質が異なる(前者は技術選定・構築作業、後者は評価・振り返り作業)ため、旧フェーズCを分割し4フェーズとする。

| フェーズ | 目的 |
|---|---|
| Ph1: 企画導入 | 認識合わせとアーキテクチャ設計のモチベーションの明確化 |
| Ph2: 設計 | 概念層〜骨格層。指標体系に基づく設計判断とサブタイプへの分岐 |
| Ph3: システム実装 | 技術選定を含む、具体設計層のIT基盤への落とし込み |
| Ph4: 実証と反復 | 検証実行・妥当性の振り返り・イテレーション |

## 0.5 適用範囲と読み方の注記(横断ルール)

以下は本ロードマップ全体、および関連ドキュメント全体に共通する解釈ルール。各項では原則としてこれを前提とし、繰り返し明記しない。

- **インフラ層の話か、協調プロトコル層の話かを逐一区別する**。「ITインフラとしてどう構築するか」(Ph3で扱う技術選定)と、「エージェント間の協調・合意メカニズムとしてどう機能するか」(Ph2の設計判断)は別問題であり、混同しない。
- **SPOF(単一障害点)への配慮は協調プロトコル層に限定する**。協調プロトコル層のSPOF(例: 単一の調停者に依存する合意設計になっていないか)はアーキテクチャ上重要な懸念事項として扱うが、インフラ層のSPOF(サーバの冗長構成、可用性設計等の非機能要件)は本アーキテクチャ構想のスコープ外とする。ITインフラの構成イメージ自体は持つが、その冗長性・可用性の設計は別の関心事として切り離す。
- **「オーケストレータ」という語も同様に層で区別する**。ITインフラ層における技術的なオーケストレーション(実行基盤・ランタイムがタスクをどう調整・実行するか)はどの実装を使っても構わず、本アーキテクチャ構想の関心事ではない。一方、**協調プロトコル層にはオーケストレータを持たない**、というのはSMASの定義上の性質そのものである。特定の主体が他の主体を中央から調整・指揮する役割(オーケストレータ)を協調プロトコル層に置いた時点で、それは集権的MAS・準集権的MASに分類されるべきものであり、SMASの前提(中央執行者なし)から外れる。
- **専門用語は初出時に定義する**。誘因構造、誘因両立性、耐戦略性、到達可能性、束(semilattice)等の理論用語は、読者にとって自明ではない。各用語が初めて登場する箇所で簡潔な定義を必ず添える。用語とその初出予定箇所の一覧は5章を参照。

## 2. Ph2: 設計

**Ph2-1. 5層モデルの提示と根拠**

3層モデル(概念層/骨格層/具体設計層)では骨格層に「手続き構造」と「誘因構造」が同居し、依拠する数理理論が異なるため設計根拠が曖昧になる。5層に分離する。

| 層 | 対応する指標 |
|---|---|
| 1. 概念層 | ①到達可能性 |
| 2. 誘因構造層 | ③④頑健性・資源コスト |
| 3. 骨格層 | ②収束性(手続き構造) |
| 4. データ構造層 | ②収束性(束構造による状態合成) |
| 5. 検証層 | ⑤検証可能性 |

分離の原則: 独立に変更・検証しうる関心事は独立した層に分離する。

**Ph2-2. SMASが暗黙に持つ前提の言語化(層ごとに)**

**Ph2-3. 指標体系①〜⑤の提示**
①到達可能性系/②収束性系/③頑健性系/④資源コスト系/⑤検証可能性系。①→⑤の順に弱い保証から強い保証へ段階的に効く構造。
(詳細は `SMAS_theorymap.md`)

**Ph2-4. 評価観点・設計項目・検証方法 + 数理手法対応表(層×指標マトリクス)**
概念層/誘因構造層/骨格層/データ構造層/検証層それぞれに対応する数理手法の一覧。誘因構造層はメカニズムデザイン基礎・VCG・契約理論・繰り返しゲーム・オークション理論を含む。
(詳細は `SMAS_theorymap.md`)

**Ph2-5. 通信モデルの選定論点**
同期/部分同期/非同期の各モデルが①の判定結果にどう効くか。
(詳細は `SMAS_theorymap.md` 1章)

**Ph2-6. 想定システム例(精緻化版)**
Ph1-6の軽い例に、指標体系①〜⑤を当てはめて精緻化する。

**Ph2-7. サブタイプ分類 → 実用的な3パターン**
指標①〜④の値の組み合わせによって定まるアーキテクチャファミリーの中から、実用的な3パターンに絞り込む。
(詳細は `SMAS_総称系とサブタイプ_提起.md`)

## 3. Ph3: システム実装

設計(Ph2)で定めた5層モデル・指標体系を、実際のITインフラ技術に落とし込む。ここが新規に厚みを持たせる部分。

**Ph3-1. 技術選定の観点**
各層は独立して技術選定しうる(Ph2-1の分離原則がそのまま技術選定の独立性にも波及する)。層ごとに「候補技術」「選定基準(指標①〜⑤のどれを優先するか)」「トレードオフ」を明示する。

**Ph3-1b. 既存MASフレームワークの評価方針**

0.5節・Ph1-4で整理した通り、既存の主要なMASフレームワークの多くは集権的MAS・準集権的MASの発想(オーケストレータの存在を前提とした設計)に立っており、協調プロトコル層にオーケストレータを持たないSMASの前提とはそもそも合致しない。したがって、既存フレームワークをそのまま協調プロトコル層の実装として採用できないケースがほとんどである、という前提に立つ。

ただし例外として**LangGraph**は、普及度の高さ(エコシステム・情報量・採用実績)から、実装基盤として使用する可能性を残す。統合にあたっては、以下の「環境層への参照プロキシ」パターンをコアルールとする(詳細は `smas_implementation_spec_for_cursor.md` 7章 を参照。本ロードマップおよび `architecture_family_map.md` とは別の実装仕様書側のドキュメント)。

**核心ルール:LangGraphのStateには「実体」ではなく「①環境層への参照」だけを持たせる**

LangGraphが標準提供するState管理機能に、各エージェントの申告内容・痕跡データの実体を直接持たせると、LangGraphのグラフ実行エンジン自体が事実上SMASの①環境層(の役割)を肩代わりする「隠れた中央集権」が発生する。SMASは「中央の執行者を置かず、環境を介した間接的な作用で協調する」設計思想であるため、これは前提の毀損にあたる。対策として、Stateには①環境層(独立したストア)への接続クライアントの参照のみを持たせ、各ノード関数は必ずそのクライアント経由でアクセスする。

統合時の確認事項(3項目):
1. 状態の実体は①環境層にあるか(Stateは参照のみを持つか)
2. 壁(アクセス制御)はフレームワークを迂回されていないか
3. フレームワークのオーケストレーション(ノード実行順序)が、誘因構造層・データ構造層・検証層(②③⑤)の役割に越境していないか

**Anti-Corruption Layer(ACL)との対立評価**: 当初「LangGraphの層を丸ごとACLでラップする」という表現をしたが、これはミスリードだった。ACLと参照プロキシは解いている問題の層が異なる。

| | ACL | 環境層への参照プロキシ |
|---|---|---|
| 解く問題 | 語彙・概念モデルの汚染(外部フレームワークの概念がドメイン語彙を侵食すること) | アーキテクチャ的性質(中央集権の不在)の毀損(フレームワークの利便機能が事実上の中央プロセスに格上げされること) |
| 効かない領域 | Stateがデータ実体を溜め込む構造上の集権化までは防げない | 語彙・概念モデルの翻訳までは統制しない |

両者は排他的でなく、組み合わせが筋が良い。**参照プロキシ(構造レベル)**でLangGraphのStateに実体を持たせないことを担保し、その内部で使う`EnvironmentClient`自体を**ACL(意味レベル)**として設計し、裏側のストア技術(KurrentDB/Redis等)の語彙をSMASのドメイン語彙(痕跡・壁のルール等)へ変換する層とする、という2層構成が有力。

なお、信頼委任システムのようなアーキタイプは状態遷移(有限状態機械)として表現できる可能性があり、その場合はLangGraphのStateGraph的な表現と構造的に相性が良い可能性がある。SMAS本体とは別に、ファミリー地図上の他アーキタイプ(信頼委任システム等)を将来的にLangGraph上へ実装する際の足がかりとして、この相性の良さは記録しておく価値がある。

**Ph3-2. 層別・候補技術マッピング(たたき台)**

| 層 | 候補技術領域 | 具体例(検討対象) |
|---|---|---|
| 概念層 | (実装対象なし、設計文書のみ) | — |
| 誘因構造層 | 分散台帳/スマートコントラクト基盤(経済的資源型)、検証可能遅延関数・PoW系(計算コスト型)、オフチェーン台帳+定期精算 | 要件確定後に選定(フェーズB4未決事項と連動) |
| 骨格層(通信・合意プロトコル) | エージェント間通信規格、オーケストレーションSDK | A2Aプロトコル、MCP、Strands Agents、LangGraph、Claude Managed Agents |
| データ構造層(束) | CRDT実装、イベントソーシング、メッセージング基盤 | KurrentDB、Redis、NATS(JetStream) |
| 検証層 | モデル検査ツール、型システム、ランタイム監視 | Quint、PRISM/Storm、TypeScript/Rustの型システム、promptfoo(評価)、Galileo Agent Control |
| ポリシー・認可(層横断) | ポリシーエンジン | AWS Cedar、OPA |

**Ph3-3. プロトタイプ対象の選定**
Ph2-7の3パターンのうち、最も実装・検証しやすいもの(例: 閉域・高信頼インフラ向けの同期モデル型)を最初の実装対象に選ぶ。理由と選定基準を明示する。

**Ph3-4. モジュール分割・実装計画**
選定した技術を踏まえ、層ごとのモジュール境界、開発順序(基盤から上位機能へ/垂直スライスで最小構成を先に通すか等)を計画する。

**Ph3-5. 開発環境・実装基盤**
WSL2、Docker、GitHub CLI等の既存ツール環境をベースに、SMAS実装用の環境構成を整理する。

## 4. Ph4: 実証と反復

**Ph4-1. 検証実行**
Ph3で実装したプロトタイプに対し、検証層の理論群(モデル検査、型検証等)を適用する。指標①〜⑤それぞれについて実測値を得る。

**Ph4-2. 妥当性の振り返り**
実測値を、Ph2で立てた設計判断・想定と突き合わせる。ズレがあれば、どの層のどの前提が原因かを特定する。

**Ph4-3. 横展開**
得られた学びを、Ph2-7の他の2サブタイプへ横展開する。層単位で再利用できる部分(例: データ構造層の実装はそのまま流用できる可能性)を明示する。

**Ph4-4. イテレーション計画**
次のサイクルで何を優先的に見直すか(層単位・指標単位)を決め、Ph2または Ph3に差し戻す。

## 5. 用語・概念解説一覧(たたき台)

設計(Ph2以降)および関連ドキュメントで使用する主要な理論用語と、簡潔な定義、初出予定箇所の対応表。

| 用語 | 簡潔な定義 | 初出予定箇所 |
|---|---|---|
| エージェント | 環境を観測し、自律的に意思決定して行動する主体。単なる処理プロセスとの違いは、状態(内部の意図・信念)を持ち、それに基づいて能動的に振る舞う点 | Ph1-2 |
| MAS(マルチエージェントシステム) | 複数のエージェントが相互作用しながら、個々もしくは全体の目的を達成しようとするシステム | Ph1-3 |
| 集権的MAS / 準集権的MAS | 単一(または部分的)のオーケストレータが他のエージェントを統括・調整する構成のMAS | Ph1-4 |
| 非協力ゲーム | 各主体が自身の効用を最大化するために独立に戦略を選ぶ(協力を強制する仕組みがない)ゲーム理論の枠組み | Ph1-4 |
| 協力ゲーム理論 | 誰と誰が組むと得か、組んだ利得をどう分けるかを扱うゲーム理論の枠組み(シャープレイ値・コア等)。非協力ゲーム理論とは異なる系統で、7ファミリーの中では提携形成システムのみが主対象とする | Ph1-4 |
| 目的関数(Objective Function) | 外部の設計者が定める、システム全体で共有された単一の評価基準。最適化理論の語彙 | Ph1-4 |
| 効用関数(Utility Function) | 各主体それぞれが内に持つ、主観的な「望ましさ」の評価基準。数式としての形は目的関数と同じだが、複数の主体がそれぞれ異なる効用関数を持ちうる点が異なる | Ph1-4 |
| 効用空間(Utility Space) | 複数主体それぞれの効用を軸として並べた多次元空間。合意点・対立点・パレート最適領域を視覚化・分析する道具 | Ph1-4 |
| 到達可能性(existence) | 与えられた通信・故障モデルの下で、目標状態への到達を保証する均衡・写像が存在するかどうか | Ph1-5 / Ph2-3 |
| 誘因構造 | 各主体が特定の行動(正直な申告、協調的な振る舞い等)を選ぶよう仕向ける、報酬・コストの設計全体 | Ph2-1 |
| 誘因両立性(インセンティブ整合性) | 各主体が自己利益を追求した結果が、設計者の望む全体の望ましい結果と自然に一致するよう設計されている性質 | Ph2-4 |
| 耐戦略性(strategy-proofness) | 各主体にとって、真の選好・情報を正直に申告することが、他者の行動によらず最適な戦略になる性質。誘因両立性の中でも最も強い保証の一つ | Ph2-4 |
| 個人合理性(IR) | 参加することで各主体が(参加しない場合と比べて)損をしない、という制約 | Ph2-4 |
| 予算均衡 | メカニズム全体で徴収する額と分配する額が釣り合っている(外部からの補填を必要としない)という制約 | Ph2-4 |
| 束(semilattice) | 任意の2元に対して最小上界(join)が常に存在する半順序集合。CRDTの収束保証の数学的基盤 | Ph2-4 |
| 収束性 | 分散した各主体の状態が、通信を重ねることで最終的に一致する状態に至る性質、およびその速さ・確率 | Ph2-3 |
| 頑健性(robustness) | 戦略的・悪意的な主体が存在しても、到達可能性・収束性がどこまで保たれるかの下限保証 | Ph2-3 |
| 検証可能性 | 設計時に主張した性質(①〜④)を、外部が事後的にどこまで確認できるかの度合い | Ph2-3 |
